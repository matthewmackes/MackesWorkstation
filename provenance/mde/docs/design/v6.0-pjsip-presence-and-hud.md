## 1. Operator locks (the four directives)

This document captures four operator-issued design locks that detail the
already-locked VOIP-* epic (Q21 of 25-Q "VoIP stays in core forever").
Each lock answers a specific architectural question that R11 left open.

### Lock 1 — PJSIP Presence wired to DND + alerting + dock control

PJSIP supports SIMPLE Presence (RFC 3856) via PUBLISH/SUBSCRIBE/NOTIFY.
Every peer's `mded` publishes its own presence and subscribes to every
other peer's presence; the resulting state drives both alerting policy
and a dock-surfaced DND control.

**Presence states:**

| State | SIP RPID | Trigger | UI render |
|---|---|---|---|
| `available` | open / activities=none | Default after login | green pip |
| `on-call` | closed / activities=on-the-phone | INVITE dialog confirmed | orange pip |
| `dnd` | closed / activities=busy | Operator toggles dock pill | red pip |
| `away` | open / activities=away | 10 min idle (mutter session-idle) | yellow pip |
| `offline` | closed / activities=unavailable | mded shutdown / SUBSCRIBE expired | grey pip |

**DND behavior:** when `self.state == "dnd"`, PJSIP's `on_incoming_call`
callback responds `486 Busy Here` immediately (mirrors the focus-filter
shape from VOIP-5). Vitelity SimRing advances; mesh peers see the
caller-side `486` and can choose retry / voicemail. No alert / ring
plays locally.

**Dock surface (Lock 1 deliverable):** a left-of-buttons pill in the
Portal dock (per [[project_v6_0_mde_portal]] R12 layout) renders the
local DND state. Click toggles `available ↔ dnd`. The pill subscribes
to Bus topic `voip/presence/<self>` so it reflects programmatic
changes too (e.g., busy-on-call auto-state).

**Bus contract:**

```
topic: voip/presence/<peer>
payload: { state: "available"|"on-call"|"dnd"|"away"|"offline",
           since: "<iso8601>",
           note: <string optional> }
retention: latest-only (last-writer-wins per peer)
```

Every UI surface that displays a peer (Portal cards, Workbench mesh
roster, mde-files mesh sidebar, mde-voice-hud peers tab, dock task
pills) subscribes to this topic for the peers it shows.

### Lock 2 — Opus default + libopus + G.711 for Vitelity

**Build:** PJSIP is built `--with-opus` so libopus is dynamically
linked. The Fedora RPM declares a hard `BuildRequires: opus-devel`
and a runtime `Requires: opus`.

**Codec priority (per-account):**

| Account | Priority order | Reason |
|---|---|---|
| `sip:<self>@mesh.mde` | opus/48000/2 ONLY | Mesh peers all carry Opus; no fallback needed |
| `sip:<sub>@sip.vitelity.net` | PCMU/8000 → PCMA/8000 | Vitelity carries G.711μ + G.711a; Opus disabled |

Configured via `pjsua_acc_modify()` with `pjsua_acc_config.codec_priority[]`
filled at account-create time. The mesh account gets `PJMEDIA_CODEC_PRIO_HIGHEST`
for Opus and explicit `PJMEDIA_CODEC_PRIO_DISABLED` for every G.711
variant. The Vitelity account inverts: PCMU + PCMA at highest, Opus
disabled. This prevents PJSIP from offering Opus on the SDP for a
Vitelity-bound INVITE (which Vitelity would reject with `488 Not
Acceptable Here`).

**Bench:** outgoing mesh INVITE's SDP shows `m=audio <port> RTP/SAVPF
111` + `a=rtpmap:111 opus/48000/2` only. Outgoing Vitelity INVITE shows
`m=audio <port> RTP/SAVP 0 8` + `a=rtpmap:0 PCMU/8000` +
`a=rtpmap:8 PCMA/8000` only.

### Lock 3 — H.264 video lock for peer-to-peer

**Supersedes** VOIP-8's R9-Q7 "VP9 default + H.264 fallback + AV1
opt-in" framing. Video codec set is **H.264 only**, no fallback to
any other format.

**Profile / level:** H.264 Constrained Baseline Profile, Level 3.1,
`profile-level-id=42e01f`, `packetization-mode=1`. Broadest hardware
acceleration support across Intel QuickSync, AMD VCE/VCN, NVIDIA
NVENC, plus software fallback via x264.

**Why H.264-only (R11-era reasoning):**

1. Hardware accel ubiquity — every PJSIP peer's GPU/iGPU has H.264
   encode + decode in silicon. VP9/AV1 hardware accel is hit-or-miss
   on the operator's 8-peer fleet.
2. SDP offer/answer simplicity — no codec negotiation surface =
   no compatibility surprises on the call-start path.
3. Future Vitelity video bridge — if/when Vitelity adds video, H.264
   is the only realistic interop codec.
4. Mediasoup bridge (VOIP-9) supports H.264 SVC natively for group
   calls; no codec re-encode needed at bridge ingress.

**Capability gate:** when initiating a video call, PJSIP advertises
H.264 only. If the answering peer's SDP doesn't include `H.264/90000`,
the dialer auto-downgrades to audio-only and surfaces a "video not
available" toast. No silent failure.

**Bench:** outgoing video INVITE's SDP `m=video` line lists payload
type for H.264 only. Negotiated parameters match `42e01f` /
packetization-mode=1.

### Lock 4 — Auto-Connect all peers at login

**On `mded` startup** (which runs at login per the v2.0.0 unified
daemon model — see `docs/design/v2.0.0-mde-rebrand/`):

1. Read peer roster from mesh-home (`<gluster_mesh_home>/voip/peers.toml`,
   GFS-replicated per [[project_v5_0_0_gluster_mesh]]) — fallback to
   `~/.config/mde/voip/peers.toml` if GFS not yet mounted.
2. PJSIP `pjsua_acc_add` mesh account (`sip:<self>@mesh.mde`) →
   REGISTER on `127.0.0.1:5060` UDP/TLS. Accept incoming.
3. PJSIP `pjsua_acc_add` Vitelity account (`sip:<sub>@sip.vitelity.net`)
   → REGISTER over TLS. Outbound trunk.
4. For every other peer in roster, `pjsua_buddy_add` + `pjsua_buddy_subscribe_pres`.
5. PUBLISH own initial presence: `available`.

**All interfaces reflect this:** every surface that lists peers
subscribes to `voip/presence/<peer>` over Bus and renders a real-time
presence pip. Specifically:

- **mde-portal Cards layer** — peer-card pips
- **mde-portal dock** — task-pill pip on every dock-pinned peer app
- **mde-workbench Mesh roster** — pip column
- **mde-files mesh sidebar** — pip next to each peer's home dir
- **mde-voice-hud Peers tab** — pip on every roster row
- **mde-portal globe** (per Portal-MESH-A) — pip on every node

**Bench (E2E):** operator A logs in. Within 8 s of the user landing
on the desktop, all 7 other peers show `registered` + correct
presence in every UI surface listed above. Operator A toggles DND
in the dock; all 7 peers' pip for A flips to red within 5 s.

## 2. mde-voice-hud crate specification

The Lock 1-4 behavior is surfaced to the operator through a
dedicated HUD crate — `crates/mde-voice-hud/` — implementing the
design imported from Claude Design bundle `wR-1e6hL49u6HQnilIjAbQ`.

### 2.1 Bundle adaptations for R11 + platform locks

The imported bundle assumes a Kamailio-routed architecture (retired
by R11) and uses Material 3 dark palettes with Roboto Flex + JetBrains
Mono fonts (the platform's locked stack is Roboto + Roboto Mono per
Q1+Q2 of 25-Q, with Material Symbols Rounded icons per Q43+Q97). The
adaptations applied during port:

| Bundle element | Platform substitute | Rationale |
|---|---|---|
| `kamailio-mde` hop pills in routing strip | DROPPED — direct PJSIP-to-Nebula hops only | R11 retired Kamailio per-peer |
| `rtpengine-mde` hop pills | DROPPED — PJSIP handles SRTP natively | R11 same |
| `kamcmd · trace call` SIP-log header | `pjsua · trace call` | Tool name matches stack |
| Roboto Flex font | Roboto (regular family) | Q1+Q2 of 25-Q lock |
| JetBrains Mono font | Roboto Mono | Q1+Q2 of 25-Q lock |
| Material Symbols Rounded icons | Material Symbols Rounded | matches platform Q43 lock — KEPT |
| Material 3 dark surface tokens (#131316 etc.) | ChromeOS Classic palette adapted (`mde-theme::voice_hud` token group) | Q1+Q2 ChromeOS lock, but voice-call surface gets its own focused dark sub-palette inside the lock |
| Mesh map "16-peer mesh" sub-text | "8-peer mesh" | Q3 + Q22 (25-Q) cap |
| Static demo peer roster `PEERS` array | Live roster from Bus topic `mesh/roster` | Auto-Connect lock 4 |
| `setTimeout` demo incoming-call after 14 s | Bus topic `voip/call/incoming` from mded | PJSIP wiring |

The routing-strip simplification under R11:

| Call kind | New hop chain |
|---|---|
| Mesh direct | `pjsip (self) → nebula1 → nebula1 (peer) → pjsip (peer)` (4 hops) |
| Mesh transit | `pjsip (self) → nebula1 → nebula1 (transit) → mediasoup-bridge → nebula1 (peer) → pjsip (peer)` (6 hops; only when VOIP-9 bridge engaged for 3+ party call) |
| Vitelity PSTN | `pjsip (self) → vitelity-edge (TLS) → vitelity-sbc → pstn` (4 hops) |

### 2.3 Bus topic contracts

**Subscribed (consumed by HUD):**

| Topic | Payload | Effect |
|---|---|---|
| `voip/presence/<peer>` | `{state, since, note?}` | Updates peer pip |
| `voip/call/<call-id>/state` | `{state: "calling"\|"ringing"\|"connected"\|"ended", codec, transit?}` | Drives HUD view transition |
| `voip/registration/<self>` | `{state: "registered"\|"failed", account}` | Updates topbar status |
| `mesh/roster` | `[{ext, name, role, lan, hint}]` | Populates Peers tab |
| `voip/link-rtt/<peer>` | `{rtt_ms, measured_at}` | Routing-strip RTT chip |

**Published (produced by HUD):**

| Topic | Payload | Effect |
|---|---|---|
| `voip/call/command` | `{op: "dial"\|"hangup"\|"accept"\|"decline", target, video?}` | mded executes via PJSIP |
| `voip/call/<call-id>/control` | `{mute?, hold?, dtmf?, transfer?}` | mded mutates active call |
| `voip/presence/<self>` | `{state, since}` | mded forwards as SIP PUBLISH to all subscribers |

### 2.4 Launch paths

- **Mod+P** keyboard shortcut — lifted into Portal-* keybinding map
  (next to Mod+r → Portal-compact-resize per R12)
- **Right-click phone-tile-in-dock** → "Open mde-voice-hud"
- **mde-portal Cards layer "VoIP" card tap** → opens HUD
- **D-Bus method** `dev.mackes.MDE.VoiceHud.Open(target?)` — for
  scripted invocation (e.g., desktop `tel:` handler)

### 2.5 Surface placement (wlr-layer-shell)

- **Layer:** `Overlay` (above normal windows; below lockscreen)
- **Anchor:** `Bottom | Right`
- **Margin:** 16 px right, 56 px bottom (clearance over dock)
- **Size:** 420 × 720 px (cozy density default); 388 × 660 (compact); 460 × 780 (comfortable)
- **Exclusive zone:** 0 (HUD doesn't push the workspace)
- **Keyboard interactivity:** `OnDemand` — gains focus on click

### 2.6 Dismissal

- **Outside-click** (clicks landing on layer-shell background) dismisses the HUD
- **Escape key** dismisses
- **Active call open?** — Escape minimizes to a floating call-pill in the dock corner; only `Hangup` fully closes the HUD
- HUD state persists across dismiss/reopen for ≤ 5 minutes (open new HUD lands you back on the previously-active tab and target)

## 3. Acceptance criteria

Each item below is bench-observable on the operator's 8-peer fleet,
verified at HW-3 / HW-4 bench passes per CLAUDE.md §0.15.

### 3.1 Lock 1 — Presence + DND + dock

- [ ] Peer A toggles DND via dock pill → peer B's Portal renders A's
      pip in red within 5 s.
- [ ] Peer C calls peer A while A is in DND → PJSIP responds `486
      Busy Here` within 200 ms; no ring plays on A.
- [ ] A goes idle (10 min mutter session-idle) → A's pip auto-flips
      to `away` across all peers within 30 s.
- [ ] A places an outgoing call → A's pip auto-flips to `on-call`;
      reverts to previous state on hangup.

### 3.2 Lock 2 — Opus + G.711

- [ ] Outgoing mesh INVITE SDP shows ONLY `opus/48000/2`.
- [ ] Outgoing Vitelity INVITE SDP shows ONLY `PCMU/8000` and
      `PCMA/8000`.
- [ ] mesh-to-mesh call's `rtpmap` matches Opus on both legs.
- [ ] mesh-to-Vitelity bridged call (via mediasoup VOIP-9) shows
      Opus on the mesh leg, PCMU on the Vitelity leg.

### 3.3 Lock 3 — H.264 video

- [ ] Outgoing video INVITE SDP lists ONLY H.264 with
      `profile-level-id=42e01f` and `packetization-mode=1`.
- [ ] Video call between two H.264-capable peers establishes within
      3 s; hardware encoder usage visible in `vainfo` / `nvidia-smi`.
- [ ] Initiating video to a peer that doesn't advertise H.264 (force
      via SDP munge for test) → HUD shows "video not available"
      toast within 2 s; call continues audio-only.

### 3.4 Lock 4 — Auto-Connect at login

- [ ] Login on peer A → within 8 s, all 7 other peers in roster
      render as `registered` + correct presence in every UI surface
      listed in §1 Lock 4.
- [ ] Bring peer B offline → peer B's pip flips to `offline` on all
      peers within 30 s (SUBSCRIBE expiry detection).
- [ ] Bring peer B back online → peer B's pip flips back to
      `available` within 10 s of B's `mded` start.
- [ ] No surface ever shows a stale presence > 60 s old without an
      explicit "stale" indicator (greyed pip + tooltip "last seen
      Xm ago").

### 3.5 mde-voice-hud surface

- [ ] `cargo build -p mde-voice-hud` exits 0; `cargo run` opens a
      layer-shell HUD anchored bottom-right.
- [ ] Mod+P from anywhere on the desktop opens the HUD with focus.
- [ ] Dialing `1003` from the keypad → HUD transitions
      idle→calling→in-call as mded fires `voip/call/<id>/state`
      events.
- [ ] Routing strip live-renders 4 hops for direct mesh, 6 hops for
      transit mesh, 4 hops for Vitelity PSTN — no Kamailio /
      rtpengine pills present.
- [ ] Mesh map overlay opens via header button; active-call edge
      animates with dashed-flow; tapping a non-self peer dismisses
      the map and starts a call to that peer.
- [ ] Outside-click dismisses HUD; Escape during in-call minimizes
      to a dock call-pill.

## 4. Risks + open questions

### 4.1 Presence-fanout cost at scale

For an 8-peer mesh, each peer SUBSCRIBEs to 7 buddies + 1 PUBLISH
per state change. That's 8×7 = 56 SUBSCRIBE dialogs + ≤56 NOTIFY
events per state change. Manageable.

Above 8 peers (post-1.0 cap-override per Q3 + cap-overrides.md),
fanout becomes N(N-1) which scales O(N²). 16 peers = 240 dialogs.
The R11 architecture has no central presence aggregator, so each
peer carries the full fanout. **Open question for post-1.0 cap
expansion:** introduce a Bus-bridged presence aggregator
(`voip/presence-aggregate` topic) so peers can SUBSCRIBE once to
the aggregate rather than per-peer. Deferred past 1.0.

### 4.2 PJSIP DTLS-SRTP interop with Vitelity

Vitelity historically accepts only plain SRTP-AES (RFC 4568, sdes
key exchange in SDP). PJSIP's default DTLS-SRTP profile won't
negotiate with Vitelity's SBC. Vitelity-account codec config must
also disable DTLS-SRTP and force `RTP/SAVP` with `a=crypto:` SDES
lines (already partially captured in VOIP-17). Verify at HW-3.

### 4.3 Auto-Connect timing on slow-mount GFS

Lock 4 reads peer roster from `<gluster_mesh_home>/voip/peers.toml`.
If GFS isn't mounted by mded-startup (cold boot, slow nebula tunnel
setup), Auto-Connect falls back to `~/.config/mde/voip/peers.toml`
(per-peer local copy). The local copy must be refreshed every time
GFS reads succeed so the fallback isn't stale. Acceptance bullet
in VOIP-26 covers this.

### 4.4 Roboto Flex vs Roboto fallback

The design bundle specifies Roboto Flex (variable axes). Q1+Q2 of
25-Q locks "Roboto" generally; Roboto Flex is the modern variable
form of Roboto and should be acceptable, but the lint config in
`data/css/tokens.css` may need a token entry. If lint rejects,
fall back to Roboto + Roboto Mono. Decided at implementation time.

## 5. Out-of-scope (deferred or rejected)

- **Mockup-only states** — every HUD state must be reachable via a
  real call OR a Bus-driven test fixture. No `setTimeout` demo
  triggers in shipped code (§0.12 no-stubs rule). The bundle's
  14-second demo-incoming-call is dropped.
- **Tweaks panel** — the bundle's right-side tweaks panel (preset /
  density / keypad style / FAB shape) is a design-prototype
  affordance, not user-facing in the shipped product. Settings
  surface via Portal's Settings layer, not the HUD itself.
- **Code-only mesh layout** — the bundle's force-laid mesh-map peer
  positions are hardcoded for 8 peers. Reasonable for 1.0 cap.
  Force-directed layout deferred past 1.0 if cap expands.
- **Multi-mesh federation** — VOIP-18 already deferred past 1.0
  per R11. The HUD doesn't render multi-mesh peers.
- **mde-voice-config crate retirement** — the existing crate
  targets Kamailio config (retired by R11). Tracked separately as
  a DEAD-* candidate; not blocking VOIP-22..VOIP-30.

## 6. Source preservation

The original design bundle (`pjsip-interface.tar.gz`, 84 KB) is
referenced by short-code `wR-1e6hL49u6HQnilIjAbQ` from Claude
Design. The HTML/CSS/JS source is **not** committed to the repo —
the design adapts to platform locks (R11 simplification, fonts,
palette), so a pixel-perfect copy of the bundle isn't the goal per
the bundle's own README. The README explicitly says: *"Match the
visual output; don't copy the prototype's internal structure unless
it happens to fit."*

Key visual reference points the implementation MUST preserve:
- 3×4 numeric keypad with letters labels (ABC, DEF, …)
- Segmented tabs at top of dialer: Keypad / Peers / Recents
- Display field with live-resolved chip showing mesh / pstn / partial / invalid
- Material 3-style FAB call button (green for ok, dim for disabled)
- Avatar with ringing animation rings
- Codec chips on calling / in-call screens
- 6-button in-call action grid (mute / keypad / speaker / video / hold / transfer)
- Routing strip horizontal hop pills with IP/port under each
- SIP-log panel with color-coded `>>` / `<<` / `==` / `**` line prefixes
- Mesh-map SVG topology with pulsing active-call edge
