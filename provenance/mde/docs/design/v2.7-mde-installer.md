## 1. Why

Post-RPM-install, the machine is *installed* but not *converged*:
`dnf install mde-core` lands the bits but never resets configs/state to a
known baseline, and stale `~/.config/mde/` from an aborted prior install
can poison the next one. On upgrade, peers drift onto different versions
silently and the operator finds out hours later when an RPC schema
mismatch surfaces. `mde-install` makes "configure" a single idempotent
command; `mde-update` coordinates fleet-wide upgrades so no peer drifts
alone.

## 4. The three confirms

1. **Typed `NUKE`** (INST-5) â€” every interactive run, before any
   destructive op. Anything but the literal `NUKE` aborts "no changes
   made."
2. **Lossy-downgrade confirm** (INST-6) â€” only when the new profile is
   lower-capability than the installed one (`/var/lib/mde/installed-profile`).
   Type the *previous* profile name. `--yes` skips it but the audit log
   gets a `WARNING: lossy downgrade from <prev> to <new>` first line.
3. **`--yes` audit log** (INST-5) â€” every non-interactive (or `--yes`)
   run writes the summary tree + peer-impact to `/var/log/mde/wipe-<ulid>.log`
   before wiping; the path is printed so the operator can `tail -F` it.

```

```

The barrier state lives entirely in the GFS-replicated intent file; the
peer roster comes from the PEERVER `peers/` dir (Â§6). `dnf upgrade`
failures still count toward the quorum so one broken repo can't stall
the fleet.
