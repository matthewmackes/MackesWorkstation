# v2.7 — Peer-data convergence substrate (INST-PEERVER / INST-PEERVER-GOSSIP)

**Locked:** 2026-05-29 via 5-Q `/plan` survey.
**Supersedes:** the original INST-PEERVER acceptance (which assumed a
non-existent gossip path + an against-direction `Fleet` D-Bus method).
**Authority:** per-epic design lock (CLAUDE.md §0.14 tier 4).

---

## Problem

`mde-update` (version-skew table, INST-9), `mde-install` peer-impact
preflight (INST-5), and INST-3b's `peer_registry` all need to know
**other peers'** state (installed `mde-core` version, liveness). But
mackesd's `nodes` table reflects only this node's enrollment/CA view —
there is **no mechanism to converge node-data across peers**. A peer
can only know its own RPM version. This substrate fills that gap.

It is **prerequisite substrate for the already-locked installer epic**,
not new feature scope — clear of the §0.16 feature lock.

## Locks (5-Q survey, 2026-05-29)

| # | Decision | Choice | Rationale |
|---|---|---|---|
| Q1 | Transport | **GFS per-peer files** `~/.mde-mesh/peers/<hostname>.json` | Node-version is slow-state; AI_GOVERNANCE §3.1 routes slow state (≤1/min) to file polling on the gluster mesh-home volume. Survives offline peers (last-known state persists on the replicated volume); no broker dependency. |
| Q2 | Write cadence | **Ride the mackesd heartbeat tick** | The heartbeat worker already runs periodically + tracks liveness. Writing the file each tick keeps `last_seen` fresh (real liveness); `mde_version` is re-detected (`rpm -q`) on startup + cached, re-checked every Nth tick. No new timer. |
| Q3 | Merge / ownership | **Own-row authority** | Each peer is the SOLE writer of its own `peers/<hostname>.json`; readers union all files (one row per hostname; the file is the row). Conflict-free by construction — no two peers write the same file. Fully centerless + flat-trust. |
| Q4 | Read path | **Read the GFS files directly** | The converged data IS the replicated dir on the local mount, so any tool reads + unions the files. No IPC / D-Bus / Bus / mackesd dependency for reads — sidesteps the §3.3 D-Bus-retirement conflict entirely (no `Fleet.list_peers` method). |
| Q5 | Nodes mirror | **Mirror into `nodes`** | mackesd's heartbeat reader also merges all peer-files into its SQLite `nodes` table (adds an `mde_version` column via migration `0012`) so existing `nodes` consumers (Workbench mesh view, health reconciler) see peer versions/liveness without each re-reading files. Files authoritative; `nodes` is a derived cache. |

## Architecture

```
each peer (mackesd heartbeat tick):
    detect local mde_version (rpm -q --qf '%{VERSION}' mde-core; cached)
    write ~/.mde-mesh/peers/<hostname>.json:
        { "hostname", "mde_version", "last_seen_ms", "health" }
            │
            ▼  (GlusterFS replicates the file to every peer)
    ~/.mde-mesh/peers/  ← every peer's file lands here on every node
            │
    ┌───────┴───────────────────────────────┐
    ▼                                        ▼
mackesd heartbeat reader            installer tools (mde-update,
  union peers/*.json →                mde-install preflight, INST-3b)
  UPDATE nodes SET mde_version,         union peers/*.json directly →
  last_heartbeat_at per hostname        version-skew table / peer-impact
  (derived cache for Workbench/health)  (no mackesd dependency)
```

- **Schema home for the peer-file type:** `crates/mackes-mesh-types/`
  (the shared types crate). Both `mackesd` and `mde-installer` link it —
  this is why `mde-installer` can read peer-data without linking the
  heavy `mackesd_core`.
- **`last_seen`:** the writer stamps `last_seen_ms` each tick; readers
  compute age against wall-clock. (Clock-skew risk noted below.)
- **Decommission:** removing a peer deletes its `peers/<hostname>.json`
  (own-row authority means the file is the peer's presence).

## Acceptance criteria (bench-observable)

Captured as the PEERVER-1..6 worklist tasks. Summary:
- A peer writes a well-formed `~/.mde-mesh/peers/<hostname>.json` on
  each heartbeat tick with the live `rpm -q mde-core` version.
- On a 2-peer mesh, each peer's `peers/` dir contains BOTH files within
  one heartbeat interval after the second peer comes online.
- `mde-update` prints a version-skew table built from the unioned files,
  with a `(!)` marker on minor skew and `(!!)` on major skew (INST-9).
- mackesd's `nodes` table shows each peer's `mde_version` after a tick.
- A peer whose file is older than the stale threshold renders as stale.

## Risks

- **GFS write churn:** writing a small JSON every heartbeat replicates
  on every tick. Mitigate: only rewrite when content changes OR cap the
  cadence (heartbeat interval is already ≥ the §3.1 ≤1/min slow-state
  budget). Liveness can ride `last_heartbeat_at` mtime instead of a
  content rewrite if churn is a problem.
- **Clock skew:** `last_seen_ms` is wall-clock from the writer; readers
  on a skewed clock misjudge age. Acceptable for a personal 8-peer mesh;
  document it.
- **Stale files for hard-dead peers:** a peer that dies without
  deleting its file lingers until the stale threshold hides it; a GC
  sweep (drop files older than N days) is a follow-up.

## Out of scope

- Real-time push (this is slow-state file polling, by design).
- A Bus query topic / `Fleet` D-Bus method (readers read files directly).
- Full anti-entropy / vector-clock merge (own-row authority makes it
  unnecessary).
- Workbench UI surfacing of peer versions (separate follow-up; the
  `nodes` mirror makes the data available).
