## 0. Why this exists

The v5.0.0 mesh filesystem is replaced wholesale: **GlusterFS out,
LizardFS in.** GlusterFS gave "every peer is an equal brick" cheaply,
but in practice its split-brain heal model, weak large-file story, and
operational opacity were a poor fit. LizardFS brings native chunking,
trash/undelete, master-arbitrated consistency (no split-brain on the
online path), per-directory quotas, and topology-aware reads.

The cost LizardFS imposes — and the central tension this survey
resolved — is that LizardFS is **master-based**: exactly one active
`mfsmaster` holds all metadata at any instant. The platform's §0
master rule was "Secure, Simple, **Centerless** Workgroup." This epic
amends that wording to **"No-Fixed-Center"** (AI_GOVERNANCE.md §0):
the fleet remains centerless in *identity and trust* — every peer is
equal, any peer can become master, failover is automatic, no node is
privileged — even though metadata has a transient single-writer. See
§3.1 and AI_GOVERNANCE.md §0 for the governing record.

---

## 1. Headline

Every Mackes Desktop Environment (MDE) peer holds every chunk of the
fleet's shared **`mesh-storage`** export (replication `goal = N`,
N = peer count). The XDG user dirs (`~/Documents`, `~/Pictures`,
`~/Music`, `~/Videos`, `~/Downloads`) ARE the mesh — they bind onto
subdirs of a single LizardFS `mfsmount` rather than living on local
disk. `~/Local/` is the off-mesh escape hatch. The Nebula overlay is
the only transport + auth boundary; LizardFS daemons speak plaintext
inside the tunnel.

Every peer runs all three LizardFS roles — **chunkserver** (stores
data), **shadow master** (live read-only metadata replica, auto-
promotable), and **client** (`mfsmount`). The active master floats
among Nebula lighthouses behind a floating overlay VIP; on failure
`mackesd` promotes a shadow and moves the VIP.

The KDC2 phone-bridge's file-share UI is removed in v5.0.0 (carried
over from the Gluster plan) — paired phones drop received files into
`~/Documents/From-<phone-name>/` and the mesh replicates from there.

## 3. Architecture

### 3.1 Role model — "No-Fixed-Center"

```
                 ┌──────────────────────────────────────────┐
                 │  Per peer (every MDE host, symmetric)     │
  nebula.service │  overlay 10.42.0.0/16                     │
  ───────────────►                                           │
                 │  mfschunkserver  ── stores every chunk    │
                 │       (goal = N)                          │
                 │  mfsmaster (shadow) ── live metadata      │
                 │       replica, auto-promotable            │
                 │  mfsmount  ── mounts mesh-storage VIP     │
                 │       at $HOME/<XDG dir>                   │
                 │                                           │
  mackesd ───────► meshfs_worker:                            │
                 │   • genesis / enroll / goal management    │
                 │   • VIP ownership + shadow promotion (HA) │
                 │   • quota apply + probe                   │
                 │   • offline write-staging + replay        │
                 │   • conflict detect + LWW resolve         │
                 │   • metadata dump into state-backup       │
                 └──────────────────────────────────────────┘

   Active master = whichever lighthouse currently owns the
   floating overlay VIP (e.g. 10.42.0.1). Any peer's shadow can
   be promoted; no node is permanently privileged.
```

**Centerless guarantee (the §0 reconciliation):** the mesh is
centerless in *identity and trust* — every peer is an equal,
fully-replicated, self-sufficient member; any peer can hold the
master role; failover is automatic; no node is special by
configuration. LizardFS's single active metadata-writer is a
*transient runtime position*, not a fixed architectural center.
This is why the §0 wording moves to "No-Fixed-Center."

### 3.2 Master HA (the VIP + promotion path)

* A floating Nebula-overlay IP (the **mesh-storage VIP**, e.g.
  `10.42.0.1`) always points at the active master. Every
  `mfsmount` and chunkserver is configured against the VIP, never
  a peer's underlay address.
* `meshfs_worker` reuses the existing QNM-Shared leader-election
  lock (the same primitive that elects the mesh leader) to decide
  which lighthouse owns the VIP + runs the active master.
* On active-master loss (VIP holder unreachable for the heartbeat
  window), the worker on the next leader promotes its local shadow
  (`mfsmaster` with the latest metadata via `mfsmetarestore`),
  claims the VIP on the overlay interface, and clients reconnect
  to the same VIP address transparently.
* Only lighthouses are VIP-eligible (master role); ordinary
  workstation peers run shadows that are promotable only if elected
  to lighthouse duty — keeps the active master on always-reachable
  nodes.

### 3.3 Storage layout

* **Chunks:** `/var/lib/mde/meshfs/chunks/` on `/` (mfschunkserver
  data dir).
* **Metadata:** `/var/lib/mde/meshfs/meta/` on `/` (master/shadow
  metadata, held in RAM + persisted here).
* **Free-space guard:** birthright + the pre-flight checker warn
  when `/var/lib/mde/meshfs` lacks `1.5 × sizeof(XDG content)`
  headroom (carried over from GF-12.2).
* **Export:** `mesh-storage`, `goal = N`. `meshfs_worker` raises the
  goal on enroll and lowers it on decommission.
* **Mount points (per user, root-owned templated systemd unit):**
  * `mesh-storage/<user>/Documents` → `/home/<user>/Documents`
  * `…/Pictures` → `/home/<user>/Pictures`
  * `…/Music`    → `/home/<user>/Music`
  * `…/Videos`   → `/home/<user>/Videos`
  * `…/Downloads`→ `/home/<user>/Downloads`
* **Local-only escape hatch:** `~/Local/` is never mesh-mounted.

### 3.4 Read path (topology-aware)

LizardFS topology labels are set per-peer by `meshfs_worker` so a
client prefers the chunk copy on its own chunkserver, falling back to
the overlay only when the local chunk is missing or healing. This
keeps reads at local-disk speed (matching Gluster's local-brick
reads) and spares Nebula bandwidth for data the peer already holds.

### 3.5 Offline read + write (the regression mitigation)

LizardFS clients normally need the master for metadata. To preserve
the offline read+write promise:

* **Reads offline:** the peer's local shadow master answers metadata
  reads (it holds the full namespace); the local chunkserver serves
  the data. A disconnected laptop browses + reads its files.
* **Writes offline:** writes land in a local staging dir
  (`/var/lib/mde/meshfs/stage/`) and `meshfs_worker` replays them to
  the active master on reconnect.
* **Replay conflicts (§3.6):** if a staged write targets a path the
  master changed meanwhile, LWW-by-mtime applies (§3.6).

> Note: a peer never promotes its own shadow to *active* while merely
> isolated (that would create split-brain metadata). Offline writes
> are staged, not master-served. Promotion happens only via the
> elected-leader HA path (§3.2).

### 3.6 Conflict model

On the **online** path there are no conflicts — the single active
master serializes all metadata ops, so Gluster's split-brain `.glusterfs/
indices/xattrop` scanning is gone entirely. Conflicts only arise at
**offline-write replay**:

* Higher `mtime` wins; the loser is renamed
  `<file>.conflict-<hostname>-<ts>` in place.
* `mde-files` shows a yellow chip on `.conflict-*` files + a
  right-click "Resolve…" handler (two-pane diff → archive loser to
  `~/Local/conflict-archive/<ts>/`). This is the existing GF-13 UI,
  reused unchanged.

### 3.7 Trash / undelete

LizardFS trash is enabled with a bounded retention (default 48h,
operator-tunable). Deleted files are recoverable from the trash for
the retention window. `mde-files` gains an "Undelete recent" affordance
backed by the LizardFS trash subtree. Scheduled CoW snapshots are
**out of scope for v5.0.0** (revisit post-cut) per Q8.

### 3.8 Daemon surface (`mackesd::meshfs_worker`)

5-second tick (mirrors `nebula_supervisor.rs`) driving:

* **Genesis:** if no master answers the VIP, become the genesis
  active master + `mfsmakesnapshot`-free create of the `mesh-storage`
  export; otherwise register the local chunkserver + shadow against
  the VIP.
* **Enroll subscription:** `EnrollmentCompleted{node_id}` → start
  chunkserver+shadow on the new peer, raise `goal` to new N.
* **Revoke subscription:** CA-revoke → mark chunkserver out
  (re-replicate to hold goal=N), drop its shadow, lower goal; if the
  revoked peer held the active master, fail the VIP over first.
* **HA tick:** own/relinquish the VIP per the leader lock; promote
  local shadow on active-master loss.
* **Quota tick (hourly):** `mfssetquota` hard cap =
  `0.8 × min(free chunkserver bytes)` on the export root; soft quota
  set lower to fire the "almost full" banner. Publish `QuotaWarning`
  on the Bus at the cap.
* **Offline replay:** drain `/var/lib/mde/meshfs/stage/` on reconnect,
  applying §3.6 LWW.
* **Metadata dump:** periodic `mfsmetadump` into the state-backup
  tarball (§3.10).

### 3.9 Bus surface (no D-Bus — Q13)

Per Q96 (EPIC-RETIRE-DBUS), the status/control surface is Bus-native:

* **Commands:** `action/meshfs/<verb>` — `resolve-conflict`,
  `undelete`, `add-peer`, `remove-peer`, `bootstrap`, `status`.
  Replies on `reply/<ulid>`.
* **Events (topic `meshfs/*`):** `peer-state-changed`,
  `conflict-detected`, `heal-completed`, `quota-warning`,
  `export-ready`, `master-failover`.

Consumers: `mde-files` (live sync-badge refresh + conflict/undelete
UI), `mde-applet-mesh-status` (per-peer in-sync / heal-pending /
offline line + active-master indicator), Workbench "Mesh Storage"
panel. No new `#[interface]` block → does not trip the §0.7 #8 D-Bus
shape lint.

### 3.10 Backup integration (subsumes NF-18.4 / GF-9)

`mackesd_state_backup` (the renamed `nebula_ca_backup`) extends its
24h tick to snapshot, into a single encrypted tarball at
`{qnm_root}/{node}/mackesd/state-backup.enc`:

* The Nebula CA bundle (NF-18.4 payload).
* `mfsmetadump` of the master metadata (namespace).
* LizardFS topology + goal config + quota config.
* Chunkserver config (paths, labels).

`mackesd state restore <bundle>` reconstructs the Nebula CA **and** the
LizardFS export config + metadata on a bare peer. Because every peer's
shadow already holds live metadata, the dump is the cold/off-mesh
belt-and-suspenders, not the primary recovery path.

## 4. Acceptance criteria (bench-observable, 3-peer fleet)

1. **Substrate.** The bundled LizardFS binaries install from the MDE
   RPM (no external repo); `systemctl is-active mfschunkserver
   mfsmaster` returns `active` on every peer; the active master is
   reachable on the mesh-storage VIP.
2. **Bootstrap + goal.** First peer creates `mesh-storage`; as peers
   enroll, `lizardfs-admin list-chunkservers` grows + the export goal
   rises to the new N.
3. **XDG mount.** `mountpoint $HOME/Documents` succeeds; touching a
   file on peer A appears on B + C within the heal window (`< 5 s`);
   `~/Local/` is unaffected.
4. **Local-read affinity.** A read of a local file on peer B is
   served from B's own chunkserver (verified via chunkserver I/O
   counters / no overlay traffic spike).
5. **Master failover.** Kill the active master; within the heartbeat
   window the VIP moves to another lighthouse, a shadow is promoted,
   and clients keep serving reads + accept writes — no remount.
6. **Offline write + replay.** Pull peer B's network; write
   `~/Documents/foo.md`; reconnect. The write replays to the master;
   if A changed `foo.md` meanwhile, every peer holds the latest-mtime
   version + `foo.md.conflict-B-<ts>.md`, and `mde-files` shows the
   chip + Resolve handler.
7. **Trash.** Delete a file; recover it via `mde-files` "Undelete"
   within the retention window; confirm it's gone after the window.
8. **Large file.** Drop a 6 GB file into `~/Videos` on A; it chunks +
   replicates to B + C with no `.mesh-stub` placeholder; streaming
   read works mid-replication.
9. **Quota.** `lizardfs-admin` reports the `0.8 × min(free)` hard cap;
   writing past it returns `EROFS`; the soft cap fires the "Mesh
   almost full" banner; Workbench highlights the limiting peer.
10. **Decommission.** `mackesd ca revoke <peer>` → within a tick the
    peer leaves the chunkserver list, chunks re-replicate to hold
    goal=N among the rest, and if it held the master the VIP failed
    over first.
11. **No KDC2 file UI.** Every share/upload/pull affordance in
    `crates/mde-kdc/` is gone; phone files arrive at
    `~/Documents/From-<phone-name>/`.
12. **Backup restore.** `mackesd state backup` on A; wipe B; install
    the v5.0.0 RPM on a bare host with B's node-id; `mackesd state
    restore <bundle>` rejoins the mesh + restores the export config.
13. **No Gluster residue.** `rpm -q glusterfs-server` returns
    not-installed; no `gluster*` binary, unit, config, or source
    reference remains in the tree (the cutover-removal step, Q25).

## 5. Cutover sequencing (Q25 — no half-states)

Per §0.12, main never carries a broken or dual FS:

1. **Build LizardFS end-to-end behind the existing Gluster code** —
   substrate (CI bundle) → `meshfs_worker` → mount units → Bus
   surface → `mde-files`/applet/Workbench UI → backup. Each MESHFS-*
   task ships complete + reachable.
2. **Final task removes GlusterFS in one commit** — deps, code,
   units, config, source references, and the GF-* design doc's
   live-path role. This is the last green step; until it lands,
   Gluster remains the operative FS so main is never FS-less.

A transition feature-flag is explicitly rejected (Q25) — two FS
implementations on main, even temporarily, violate §0 "Simple."

## 6. Out-of-scope (deferred past v5.0.0)

* **Scheduled CoW snapshots / point-in-time browse** (Q8) — trash
  ships; snapshots are post-cut.
* **Erasure-coding goals** (Q4) — `goal = N` full replication only;
  EC is a later space-efficiency option.
* **Per-user quota trees** — the cap is fleet-wide against
  `min(free)`.
* **Encryption at rest** — LizardFS plaintext on disk + on the wire
  (Nebula is the boundary). LUKS at the brick block-device is an
  operator install-time choice.

## 7. Risks

* **Upstream liveness.** The original `lizardfs/lizardfs` went quiet
  ~2019; the actively-developed successor by the original team is
  **SaunaFS** (`saunafs/saunafs`). This epic honors the operator's
  explicit "LizardFS" choice and pins a maintained LizardFS fork
  commit (Q11). **If long-term maintenance matters, SaunaFS is the
  drop-in modern successor** — flagged here for an operator decision;
  the FS-agnostic `meshfs` naming (Q12) makes a later SaunaFS pivot
  near-zero-churn.
* **Single active master = metadata SPOF on the online path.**
  Mitigated by shadows-everywhere + VIP failover (§3.2) + the
  metadata dump (§3.10). Failover latency is the operative risk;
  bench item #5 measures it.
* **Master metadata RAM growth.** `mfsmaster` holds the namespace in
  RAM (~300 B/inode). For an 8-peer personal workgroup this is
  negligible (millions of files = low hundreds of MB), but the
  free-space/RAM guard (§3.3) surfaces it.
* **Offline-write replay edge cases.** Staged writes that touch many
  files or large files lengthen replay; the conflict path (§3.6) is
  the safety valve. Bench item #6 exercises it.
* **CI build fragility.** Bundling from source (Q11) means a fork
  build break blocks the cut. Pinning a known-good commit + caching
  the build artifact mitigates.
