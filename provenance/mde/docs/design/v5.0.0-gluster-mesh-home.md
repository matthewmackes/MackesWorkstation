# v5.0.0 — GlusterFS `mesh-home` (primary file sync)

**Status:** Locked 2026-05-24 via 25-Q survey
**Target release:** v5.0.0 (SemVer major — KDC2 file-transfer
removal is a user-visible breaking change for paired phones)
**Depends on:** v2.5 Nebula fabric live on every target peer
**Worklist:** `docs/PROJECT_WORKLIST.md` § "GF-1..GF-15:
v5.0.0 — GlusterFS mesh-home"

## 1. Headline

Every Mackes Desktop Environment (MDE) peer holds every file in
the fleet's shared `mesh-home` volume. The XDG user dirs
(`~/Documents`, `~/Pictures`, `~/Music`, `~/Videos`,
`~/Downloads`) ARE the mesh — they are FUSE-mounted onto the
GlusterFS-replicated brick rather than living on the peer's
local disk. Files written on any peer appear on every other
peer within the LAN-first heal window. The Nebula overlay is
the only transport + auth boundary; glusterd runs plaintext
inside the tunnel.

The KDC2 phone-bridge's file-share UI is removed entirely in
v5.0.0 — paired phones drop received files into
`~/Documents/From-<phone-name>/` and the mesh handles
replication from there.

## 3. Architecture

### 3.1 Process model

```
                          ┌────────────────────────────────┐
                          │  Per peer (any Mackes host)    │
                          │                                │
  nebula.service ────────►│ overlay 10.42.0.0/16           │
                          │   ↑                            │
                          │   │ glusterd binds here only   │
  glusterd ───────────────┘   │ (GF-1.3 vol drop-in)       │
                              │                            │
  mackesd ──── gluster_worker │ — bootstraps mesh-home,    │
                  ↑           │   probes peers, polls heal │
                  │ enroll +  │   queue, resolves LWW,     │
                  │ revoke    │   manages quota.           │
                  │ events    │                            │
                              │                            │
  per-user systemd unit ──────► mde-mesh-mount@%i.service  │
                              │   FUSE-mounts mesh-home    │
                              │   at $HOME/<XDG dir>       │
                          └────────────────────────────────┘
```

### 3.2 Storage layout

* **Brick:** `/var/lib/gluster/bricks/mesh-home/` on every peer.
* **Volume:** `mesh-home`, `replica = N` where N = peer count.
  glusterd promotes/demotes replicas as peers enrol/leave.
* **Mount points (per user, root-owned systemd unit
  templated by username):**
  * `mesh-home/<user>/Documents` → `/home/<user>/Documents`
  * `mesh-home/<user>/Pictures`  → `/home/<user>/Pictures`
  * `mesh-home/<user>/Music`     → `/home/<user>/Music`
  * `mesh-home/<user>/Videos`    → `/home/<user>/Videos`
  * `mesh-home/<user>/Downloads` → `/home/<user>/Downloads`
* **Local-only escape hatch:** `~/Local/` is never mesh-
  mounted. Operators stash one-off files here when they don't
  want fleet replication.

### 3.3 UID/GID lock

Every primary account on every peer is `uid:gid 1000:1000`.
Without this, GlusterFS file ownership diverges across peers
and FUSE shows "I don't own this" for files the local user
actually wrote. Birthright (`GF-3.1`) enforces the lock on
install — if the primary account isn't 1000:1000, it runs
`usermod -u 1000 -g 1000` + `chown -R` over `$HOME` and
`/var/lib/<user>` before the gluster bootstrap step fires.

### 3.4 Daemon surface (`mackesd::gluster_worker`)

5-second tick driving:

* **Boot:** if no peer in the Nebula overlay has the volume
  (probed via `gluster pool list` against every reachable
  lighthouse + peer), create `mesh-home` as a single-brick
  replica volume; otherwise `gluster peer probe <lighthouse-
  overlay-ip>` + `gluster volume add-brick mesh-home replica N+1
  <self-overlay-ip>:<brick-path>`.
* **Enroll subscription:** `nebula_supervisor::Enrollment
  Completed{node_id}` → `gluster peer probe <peer-overlay-ip>` +
  `add-brick`.
* **Revoke subscription:** CA-revoke → unconditional
  `gluster peer detach <peer-overlay-ip>` (auto-shrink, Q15).
* **Quota tick (hourly):** `gluster volume quota mesh-home
  limit-usage / $((0.8 * min_free_brick_bytes))`. Emit
  `QuotaWarning` signal at the cap.
* **Conflict detector:** walk every brick's
  `.glusterfs/indices/xattrop/` for split-brain entries.
  Emit `ConflictDetected{path, peers}` for each.
* **Conflict resolver (LWW):** highest-mtime version wins;
  every loser gets renamed `<file>.conflict-<hostname>-<ts>`
  in-place. Emit `HealCompleted{path}`.
* **Write watcher (5 GB cap):** any file written that exceeds
  5 GiB gets replaced by a `.mesh-stub` placeholder
  containing `{origin_host, size, sha256, gfid}` JSON. The
  real bytes stay on the originating peer; other peers
  fetch on demand via `mackesd gluster fetch-stub <path>`.

### 3.5 D-Bus surface (`dev.mackes.MDE.Gluster.Status`)

* **Service:** `org.mackes.mackesd` (same as the other five
  mackesd D-Bus services).
* **Path:** `/dev/mackes/MDE/Gluster/Status`
* **Methods:** `Status()`, `ListPeers()`, `AddPeer(node_id)`,
  `RemovePeer(node_id)`, `ConflictList()`, `HealStatus()`,
  `MountStatus()`, `BootstrapVolume()`.
* **Signals:** `PeerStateChanged`, `ConflictDetected`,
  `HealCompleted`, `QuotaWarning`, `VolumeReady`.

Consumers: `mde-files` (subscribes for live sync-badge
refresh + conflict UI), `mde-applet-mesh-status` (subscribes
for the per-peer in-sync / heal-pending / offline status
line), Workbench "Mesh Storage" panel via `Gio.DBusProxy`.

### 3.6 Conflict resolution model

Inevitable for replicated user dirs without coordination
(two peers offline, both edit `~/Documents/notes.md`, both
reconnect). Lock 8 says: **last-write-wins by mtime**, every
loser gets renamed `<file>.conflict-<hostname>-<ts>` in-place,
and the operator can resolve via mde-files's right-click
"Resolve…" handler (GF-13.1) which presents a two-pane diff
and moves the loser to `~/Local/conflict-archive/<ts>/`
once the user picks a winner.

No version history (Q17) — the conflict is the lever; once
resolved the loser is archived locally and the volume returns
to single-version-per-path.

### 3.7 Pre-mesh content migration

On first enroll, GF-3.3's `apply_xdg_mesh_mount()`:

1. For each XDG dir (`Documents`, `Pictures`, `Music`,
   `Videos`, `Downloads`), move existing content to
   `~/Local/pre-mesh-<ts>/<dirname>/`.
2. Mount the mesh-home volume at the XDG dirs (per-user
   systemd unit `mde-mesh-mount@.service`).
3. `rsync --ignore-existing` the archived content back into
   the mesh-mounted dirs.

Conflicts during step 3 (file already exists in the mesh
because another peer published it first) follow §3.6 —
the local copy becomes `<file>.conflict-<hostname>-<ts>`.

### 3.8 Backup integration (subsumes NF-18.4)

GF-9.x renames `crates/mackesd/src/workers/nebula_ca_backup.rs`
→ `mackesd_state_backup.rs` and extends the 24-hour tick to
snapshot:

* The existing Nebula CA bundle (NF-18.4 payload).
* `gluster volume info --xml` (volume topology).
* `gluster peer status --xml` (peer roster).
* Brick xattr config (mount options, perms).

Single encrypted tarball at
`{qnm_root}/{node}/mackesd/state-backup.enc` (per Q25). New
CLI `mackesd state restore <bundle>` reconstructs both the
Nebula CA and the Gluster volume config on a bare peer.

## 5. Acceptance criteria (bench-observable)

The v5.0.0 cut is greenlit when each of these passes on a
three-peer bench fleet:

1. **Substrate.** `rpm -q glusterfs-server glusterfs-fuse`
   returns installed on every peer; `systemctl is-active
   glusterd` returns `active`; `gluster pool list` shows
   all three peers in the `Connected` state.
2. **Bootstrap.** On the first peer enrol, `mesh-home`
   appears in `gluster volume list` with one brick. As each
   subsequent peer enrols, the brick count grows + `gluster
   volume info mesh-home` reports the new replica count.
3. **XDG mount.** On every peer, `mountpoint $HOME/Documents`
   succeeds; `df -h $HOME/Documents` reports the mesh-home
   volume as the backing store; touching a file in
   `~/Documents` on peer A shows up under `~/Documents` on
   peers B + C within the LAN-first heal window (target
   `< 5 s`).
4. **Conflict resolution.** Pull peer B's network, edit
   `~/Documents/foo.md` on both A and B; reconnect. Within
   30 s every peer holds `foo.md` (latest mtime) +
   `foo.md.conflict-B-<ts>.md` (loser). mde-files shows a
   yellow chip on the conflict file + the right-click
   "Resolve…" handler opens.
5. **Stub fall-back.** Drop a 6 GB file into `~/Videos` on
   peer A. Within 60 s, peers B + C see
   `~/Videos/<filename>.mesh-stub` (the placeholder JSON);
   right-clicking → "Fetch from peer-A" downloads the real
   bytes via `mackesd gluster fetch-stub`.
6. **Phone share.** Pair a phone to peer A via KDE Connect.
   Push a file from the phone; observe it at
   `~/Documents/From-<phone-name>/<file>` on peer A within
   2 seconds, then on peers B + C within the LAN-first heal
   window.
7. **Quota.** `gluster volume quota mesh-home list` reports
   the configured `0.8 × min(free brick)` cap; writing past
   the cap returns `EROFS`; mde-files surfaces a persistent
   "Mesh almost full" banner; Workbench Mesh Storage panel
   highlights the limiting peer in red.
8. **Decommission.** `mackesd ca revoke <peer-node-id>` on
   any peer → within one tick, the revoked peer disappears
   from `gluster peer status` on every remaining peer + the
   volume's replica count decreases by one.
9. **No KDC2 file UI.** Every share / upload / pull button
   in `crates/mde-kdc/` has been removed; clicking the
   phone's peer card in `mde-peer-card` shows no file-share
   affordance. The phone's KDE Connect app reports the peer
   as "no file sharing".
10. **Backup restore.** Take a `mackesd state backup` on
    peer A; wipe peer B; install the v5.0.0 RPM on a bare
    Fedora 44 host with peer B's node-id; run `mackesd
    state restore <bundle>`; observe that peer B rejoins
    the mesh AND the Gluster volume comes back with its
    original peer position + brick path.

## 6. Out-of-scope (deferred)

* **Version history / snapshots.** Q17 explicitly locks
  "live state only." If the operator wants snapshot history,
  it lands as a separate v5.1 feature using GlusterFS's
  native `snapshot` subcommand — not at v5.0.0.
* **Tiered storage / SSD cache.** Every peer has the full
  fleet. No hot/cold partitioning.
* **Quotas per-user.** Q16's cap is fleet-wide, set against
  `min(free brick)`. Per-user limits would require an extra
  GlusterFS quota tree — deferred.
* **Encryption at rest.** GlusterFS plaintext on disk + on
  the wire (Q3 — Nebula is the auth/transport boundary,
  not a second TLS layer). LUKS at the underlying brick
  block-device is out of scope for v5.0.0; operators who
  want it provision LUKS at install time before the brick
  filesystem mounts.

## 7. Risks

* **Brick filling up unexpectedly fast.** The 5 GB stub cap
  bounds per-file growth but the fleet quota cap is what
  prevents catastrophic fill. The hourly probe in GF-2.7 is
  the primary signal; the EROFS cap is the hard stop.
* **Split-brain feedback loops.** The LWW resolver renames
  losers in-place; if two peers race the rename, both
  `.conflict-*` siblings get rewritten. Mitigated by
  glusterd's own write-ordering inside the tunnel — every
  mtime+rename pair is atomic at the brick level.
* **GlusterFS daemon crash recovery.** glusterd respawn is
  handled by systemd's `Restart=on-failure` (shipped by the
  upstream `glusterfs-server` package). The gluster_worker
  also notices `gluster pool list` failures + emits a
  `QuotaWarning` shape with reason `daemon-unreachable` so
  the panel surfaces the problem.

## 8. Migration from v4.0.x

* **Upgrade RPM.** dnf pulls in the new `glusterfs-server`
  + `glusterfs-fuse` deps; `%post` enables glusterd.
* **Existing XDG content.** GF-3.3 archives existing
  content to `~/Local/pre-mesh-<ts>/` then `rsync --ignore-
  existing` back into the mesh-mounted dirs. Operators
  curating their own `~/Documents/` layout can move files
  from `~/Local/pre-mesh-<ts>/` back into the mesh after
  install.
* **Paired phones.** KDE Connect's file-share affordance
  goes dark in v5.0.0. Phones drop files into the mesh
  drop folder instead. Phone-side UX docs (GF-15.1)
  cover the change.
* **Pre-flight checker (GF-12.2).** Workbench warns
  (doesn't block) when
  `/var/lib/gluster/bricks` doesn't have at least
  `1.5 × sizeof(XDG content)` free, so the brick has
  headroom for replication overhead.
