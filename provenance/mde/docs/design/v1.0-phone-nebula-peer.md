## 0. Why elevate the phone?

Through v5.0.0 the phone sat "beside the mesh": KDC2 plugins
synced notifications + SMS + clipboard + battery + mpris over
the legacy KDE-Connect-derived TLS-over-WAN channel, but the
phone was **never a Nebula peer**. The 8-peer Q22 cap counted
desktops only; the phone was an asymmetric appendage.

Q23 of the 25-Q tuning survey (2026-05-26) reopened Q58 of the
100-Q tightening survey: the phone gets **full Nebula peer-hood
with limited resource access**. Bus + Nebula service access
become symmetric with desktops; GFS access stays scoped (no
FUSE-on-Android — see §6).

This brings cross-device continuity into the same flat-trust
mesh model as desktop peers: the phone's audit-log entries
join the same Bus topic tree, the operator's clipboard + SMS +
notifications flow over the same Nebula overlay (not a parallel
TLS channel), and the phone's `fdo/#` subscription receives
cross-peer notifications natively.

## 1. Architecture

| Layer | Phone role |
|---|---|
| **Nebula overlay** | Full peer with mesh-CA-signed cert; participates in the lighthouse discovery cycle; reachable at `<phone-hostname>.mesh.mde` |
| **Bus (mde-bus)** | Publishes + subscribes against the per-peer ntfy broker over Nebula; uses `mde-bus` binary OR a shell-out-equivalent (§7) |
| **KDC2** | **Stays** as the battery-aware optimization layer for the 5 surviving plugins (clipboard + SMS + battery + mpris + ping/find-my-phone); supersedes neither Bus nor Nebula but layers on top of them for battery-friendly push semantics |
| **GFS (mesh-home)** | **Read-only drop-folder access only.** No FUSE-on-Android (§6). Phone writes via the existing `~/Documents/From-<phone>/` drop folder (per `[[project_kdc2_file_transfer_removed]]`) which then replicates via desktops' FUSE mount |
| **Voice/Vitelity SIP** | Out of scope for TUNE-16; phone uses the operator's existing carrier app for cellular voice. Mesh-internal VOIP-* PJSIP integration deferred to a separate post-1.0 epic |

The phone runs an Android app that bundles:

- The **Nebula Android client** (the official open-source one,
  vendored to a tag pinned in the build pipeline).
- The **KDC2 protocol stack** (existing).
- A **`mde-bus` shell-out replacement** that speaks ntfy
  pub/sub directly + provides the same CLI surface the
  `mde-bus` binary does on desktops (mint passcodes, publish,
  subscribe, status).

## 2. Pairing flow

Per the Q23 R1 risk-mitigation choice, pairing extends
Birthright with a phone-pair step:

1. Operator launches **Workbench → Mesh → Devices → Add
   phone** on a desktop peer.
2. Workbench mints a Nebula client cert signed by the mesh CA
   (via the existing nebula-supervisor's CA worker) with the
   phone's hostname (operator-typed; defaults to a
   `<operator-name>-phone` template).
3. Workbench renders a QR code encoding `{nebula-config,
   mesh-ca-fingerprint, phone-cert-pem, lighthouse-endpoints,
   passcode-envelope}`.
4. Operator opens the MDE Android app + scans the QR code.
5. The app validates the passcode envelope (single-use,
   24-hour expiry, mesh-CA-bound — same shape as the
   federation passcode envelope from
   `docs/design/v1.0-federation-pairing.md` §1, modulo
   recipient-scope).
6. The app installs the Nebula config + starts the overlay
   + spawns the `mde-bus` shell-out + KDC2 stack.
7. The phone announces itself on Bus topic
   `mesh/peers/joined` and shows up as a peer in every
   desktop's Peer Card list.

Re-pairing (cert expiry / phone reset) reuses the same flow;
the old cert is revoked by the CA worker on successful new
pair.

## 3. Q22 8-peer cap

The phone **counts** against the Q22 8-peer cap. A typical
deployment of 4 desktops + 1 phone = 5 of 8 used. Q22's
reasoning (cap exists to keep flat trust + mesh routing
tractable) applies symmetrically to phones; the phone is
indistinguishable from a desktop at the Nebula + Bus layer.

This differs from federation peers (`docs/design/v1.0-
federation-pairing.md` §6) which are explicitly *not* counted
because federation is an inter-mesh bridge, not membership.

## 4. Bus pub/sub from the phone

The Android app's `mde-bus` shell-out provides:

- `publish <topic> <body> [--priority ...]` — write to ntfy
  broker over Nebula
- `subscribe <topic-pattern>` — long-polling subscribe;
  results pumped into the OS notification surface
- `status` — list pairs, last-seen-broker timestamp, queue depth

**Default subscriptions** the app maintains:

- `fdo/#` — every desktop's FDO notification (cross-peer
  awareness)
- `voip/incoming/*` — surface incoming calls on phone (per
  the VOIP-5 SimRing + PJSIP focus filter — once VOIP-* lands,
  phone joins SimRing as another sub-account)
- `mon/critical/*` — Netdata critical alerts surface as
  high-priority phone notifications

**Phone-side publish** is limited to: the operator's
explicit clipboard send (KDC2 path), SMS bridge (KDC2 path),
operator-initiated mesh-control commands (`mde-bus publish
action/portal/lock` etc.). Per the flat-trust mesh model the
phone CAN publish anywhere it has shell-out capability for;
the practical limit is what the Android app surfaces in its
UI.

## 5. KDC2 stays as battery-aware optimization

The KDC2 protocol stack continues to drive the 5 surviving
plugins (clipboard / SMS / battery / mpris / ping). Reasons it
stays alongside Bus rather than getting subsumed:

- **Battery semantics:** KDC2's push semantics over LAN
  (when reachable) avoid spinning up Nebula tunnels for
  short interactions. Bus over Nebula is the canonical fall-
  through; KDC2-on-LAN is the warm path.
- **SMS bridge:** KDC2's SMS plugin already federates with
  Android's SmsManager + Conversations bridge; replicating
  that in `mde-bus` would re-implement an existing surface
  for no win.
- **mpris:** Same — KDC2's media plugin is mature; the Bus
  side would re-wrap it.

The phone runs both layers in parallel. The operator never
sees the layering — they see "phone notifications appear on
desktop" + "desktop clipboard appears on phone"; the routing
layer is transparent.

## 6. GFS access — drop-folder only

**No FUSE-on-Android.** The phone does NOT mount mesh-home.
File transfers happen through the existing drop-folder
pattern (`[[project_kdc2_file_transfer_removed]]`):

- Phone → desktop: phone's KDC2 file-share writes to
  `~/Documents/From-<phone-name>/` on the receiving desktop,
  which then replicates via the desktop's FUSE mount.
- Desktop → phone: drag a file onto the Peer Card; KDC2's
  share plugin pushes it via the existing TLS path.

Rationale: FUSE-on-Android requires either rooting the device
or shipping a privileged service; both are operator-hostile
(rooting breaks Play Protect; a privileged service breaks
sandboxing). The drop-folder pattern is fully functional
for cross-device file flow without those costs.

**Not "deferred to 1.1" — out of scope entirely for
TUNE-16.** Per `[[feedback_1_0_is_the_whole_budget]]` /
`[[feedback_1_0_is_the_whole_backlog]]` the platform doesn't
hold features for future feature releases; FUSE-on-Android
isn't a feature being held — it's a path we've explicitly
chosen not to take. v1.1 (bug-fix follow-up) won't add it
either.

## 7. Voice-and-tone

Phone appears in MDE UI labeled **"Mesh peer (phone)"** —
never "Phone" alone (the "phone" prefix can collide with the
PSTN voice surface labels under VOIP-*) and never "Phone
peer" (subtly grammatically jarring).

The Peer Card render gains a phone-glyph variant (small
mobile-device icon overlaid on the standard peer dot) +
"Mesh peer (phone)" subtitle.

   message that lands on every desktop + receives a Bus

   typed per §0.15 + 25-Q Q14).

## 9. Risks

| Risk | Mitigation |
|---|---|
| Phone leaks mesh CA cert when lost/stolen | `mde-bus federation revoke <phone>` (or the equivalent mesh-internal `mde peer revoke <phone>`) immediately removes the cert + emits revoke event on Bus; remote-wipe via KDC2 find-my-phone if still reachable |
| Phone-side `mde-bus` shell-out drifts behind desktop binary | Pin the Android app's protocol version to a specific Bus schema tag (`mde-bus --schema-version` reports it); broker rejects connections with incompatible schema; app prompts operator to update |
| Battery drain from Nebula always-on | Nebula's UDP idle is low; Android's Doze + App Standby will background it within minutes of inactivity; KDC2 LAN-warm-path handles short-interaction notifications without Nebula spinup |
| Phone subscribes to noisy topic + drains battery | Default subscription set (§4) is conservative; operator can add subscriptions explicitly via the app's settings + the subscription list is visible per-topic with battery-usage estimate |
| Lighthouse unreachable from cellular | The phone retries via the lighthouse's HTTPS-443 tunnel (NF-1.x), mirroring the desktop fallback path |

## 10. Out-of-scope (v1.0)

- **FUSE-on-Android.** Discussed in §6; explicitly not a
  future-release-deferred feature.
- **Phone as VOIP PJSIP peer.** The phone uses the
  operator's existing cellular carrier app for voice; mesh-
  internal SimRing integration handled per VOIP-5 once
  PJSIP/Vitelity ships; full PJSIP-on-Android deferred to a
  post-1.0 epic only because of mobile-OS audio-stack
  complexity, not as a feature held for 1.1.
- **iOS support.** Per AI_GOVERNANCE.md Q40 (Android only,
  iOS explicitly unsupported).
