# v1.0 — Iced 0.13 → 0.14 + iced_layershell 0.13.7 → 0.18 migration guide

**Locked:** 2026-05-27 via UX-PRE.a-migration (split per §0.12 from UX-PRE)
**Authority:** UX-PRE worklist task + this document.
**Status:** Conversion patterns locked; per-crate ports lift under UX-PRE.b/.c/.d/.e/.f.
**Memory:** `[[feedback_rustup_pending]]`, `[[feedback_1_0_is_the_whole_backlog]]`.

## 0. Context

The 2026-05-23 UX-PRE re-probe attributed the iced 0.14 bump to
upstream `softbuffer` + `winit` releases that never shipped. The
2026-05-27 ship-CH worktree probe found that hypothesis is
wrong — iced 0.14.0 + iced_layershell 0.18.1 compile cleanly on
both system rustc 1.95 and rustup-managed 1.94 against
softbuffer 0.4.8 + winit 0.30.13. The actual port is a
workspace-wide consumer-code API migration: ~200 errors across 5
crates, mechanical patterns documented below.

This document is the **conversion guide** every per-crate port
references. The patterns are the same across crates; what
varies is which patterns each crate happens to use.

Quick error-count map per crate (from the ship-CH `cargo check
--workspace --all-targets --keep-going` enumeration):

| Crate | Errors | Sub-task |
|---|---|---|

| `mde-popover` | 57 | UX-PRE.c |
| `mde-voice-hud` | 20 | UX-PRE.d |

## 1. Cargo.toml: deps bump + new feature flag

Every consumer crate's iced dep changes:

    # Before (iced 0.13):
    iced            = { version = "0.13", default-features = false,
                        features = ["wgpu", "tiny-skia", "svg",
                                    "tokio", "advanced"] }

    # After (iced 0.14):
    iced            = { version = "0.14", default-features = false,
                        features = ["wgpu", "tiny-skia", "svg",
                                    "tokio", "advanced", "wayland"] }

**Critical:** iced 0.14 dropped default inclusion of the
`wayland` / `x11` backend features. Without explicitly enabling
one, you get:

    error: No Unix display server backend has been enabled. You
           must enable a display server feature.
           Available options: x11, wayland.

The platform is Wayland-only per the v6.5 Hyprland lock, so
every iced consumer adds the `wayland` feature.

For `iced_layershell`:

    # Before (0.13.7):
    iced_layershell = "0.13.7"

    # After (0.18):
    iced_layershell = "0.18"

For crates that only use `iced_core` (e.g. `mde-theme`):

    # Before:
    iced_core = { version = "0.13", optional = true }

    # After:
    iced_core = { version = "0.14", optional = true }

## 2. iced_layershell::Application trait → builder pattern

The largest single API change. iced 0.14 + iced_layershell 0.18
both moved from trait-based `Application` to a builder pattern
that mirrors iced's own. **128 errors** in the workspace come
from this single change.

### Before (iced_layershell 0.13.7)

    impl iced_layershell::Application for App {
        type Executor = iced::executor::Default;
        type Message = Message;
        type Theme = Theme;
        type Flags = ();

        fn new(_flags: ()) -> (Self, Task<Message>) {
            (App::default(), Task::none())
        }

        fn namespace(&self) -> String {
            "mde-portal".to_string()
        }

        fn update(&mut self, msg: Message) -> Task<Message> {
            // ...
        }

        fn view(&self) -> Element<'_, Message> {
            // ...
        }

        fn subscription(&self) -> Subscription<Message> {
            // ...
        }

        fn theme(&self) -> Theme {
            mde_theme::default_theme()
        }

        fn style(&self, theme: &Theme) -> Appearance {
            // ...
        }
    }

    pub fn run() -> iced_layershell::Result {
        <App as iced_layershell::Application>::run(Settings {
            id: Some(APP_ID.to_string()),
            layer_settings: LayerShellSettings { ... },
            ..Default::default()
        })
    }

### After (iced_layershell 0.18)

    use iced_layershell::application;

    pub fn run() -> Result<(), iced_layershell::Error> {
        application(App::default, namespace, update, view)
            .style(style)
            .subscription(subscription)
            .theme(theme)
            .settings(Settings {
                layer_settings: LayerShellSettings { ... },
                ..Default::default()
            })
            .run()
    }

    fn namespace(_state: &App) -> String {
        "mde-portal".to_string()
    }

    fn update(state: &mut App, msg: Message) -> Task<Message> {
        // ...
    }

    fn view(state: &App) -> Element<'_, Message> {
        // ...
    }

    fn subscription(_state: &App) -> Subscription<Message> {
        // ...
    }

    fn theme(_state: &App) -> Theme {
        mde_theme::default_theme()
    }

    fn style(state: &App, theme: &Theme) -> Appearance {
        // ...
    }

### Conversion checklist per crate

1. Remove `impl iced_layershell::Application for App { ... }`
   block entirely.
2. Convert each trait method (`new`, `namespace`, `update`,
   `view`, `subscription`, `theme`, `style`) into a free
   function taking `&App` / `&mut App` as its first arg.
3. Move type aliases (`type Message = ...`, `type Theme = ...`)
   to wherever they're already exported, since iced 0.14 infers
   them from the closures.
4. Rewrite `run()`:
   - `<App as iced_layershell::Application>::run(Settings { ... })`
   - → `application(App::default, namespace, update, view).settings(Settings { ... }).run()`
5. The `#[to_layer_message]` attribute on `Message` is
   preserved — it works identically on 0.18.

## 3. `iced::widget::Space` API rewrite

The `Space` widget API changed from arg-bearing constructor +
named alternates to a 0-arg constructor + builder methods.

### Before

    use iced::widget::Space;

    Space::new(Length::Fixed(8.0), Length::Fixed(8.0))
    Space::with_width(Length::Fill)
    Space::with_height(Length::Fixed(12.0))

### After

    use iced::widget::Space;

    Space::new().width(8.0).height(8.0)
    Space::new().width(Length::Fill)
    Space::new().height(12.0)

Or via the helper at `iced::widget::space()`:

    use iced::widget::space;
    space().width(Length::Fill)

Note that `Space::new()` now accepts numeric literals via
`Length: From<f32>` — `Length::Fixed(8.0)` becomes just `8.0`.

### Error patterns

- **26 errors**: `Space::new` "takes 0 arguments but 2 were
  supplied" — convert to chained builder methods.
- **19 errors**: `with_width` / `with_height` "no function
  found" — convert to `.width(...)` / `.height(...)` on a
  `Space::new()`.

## 4. `iced::widget::horizontal_space` rename

Renamed to plain `space`:

### Before

    iced::widget::horizontal_space()

### After

    iced::widget::space()

5 errors in the workspace; mechanical search-and-replace.

## 5. Keyboard event API: `on_key_press` → `event::listen_with`

`iced::keyboard::on_key_press` is gone. Replace with a generic
event listener that filters for keyboard events.

### Before

    iced::keyboard::on_key_press(|key, modifiers| match key {
        iced::keyboard::Key::Named(Named::Escape) => {
            Some(Message::EscapePressed)
        }
        _ => None,
    })

### After

    use iced::event;
    use iced::keyboard;

    event::listen_with(|event, status, _window| match event {
        iced::Event::Keyboard(
            keyboard::Event::KeyPressed { key, modifiers, .. }
        ) if status == event::Status::Ignored => {
            match key {
                keyboard::Key::Named(keyboard::key::Named::Escape) => {
                    Some(Message::EscapePressed)
                }
                _ => None,
            }
        }
        _ => None,
    })

17 errors in the workspace; the conversion is mechanical but
the callback signature widens (3 args instead of 2).

## 6. Closure signature changes (subscriptions + button)

Several callbacks widened by one parameter in iced 0.14. The
compiler reports them as:

    error: closure is expected to take 0 arguments, but it takes 1 argument

Or vice versa. Patterns:

- `Subscription::run(|| ...)` now passes a `&State` if the
  application is generic over state. Add the parameter.
- `button(...).on_press(|...| ...)` callbacks may have changed
  signature; check the error message for the expected arity.

## 7. New required struct fields

iced 0.14 added fields to two public structs that must be
explicitly initialized.

### `iced::widget::container::Style`

New `snap: bool` field. Default is `false`. Patch every literal:

    // Before:
    container::Style {
        background: Some(Color::WHITE.into()),
        border: Border::default(),
        ..Default::default()
    }

    // After:
    container::Style {
        background: Some(Color::WHITE.into()),
        border: Border::default(),
        snap: false,
        ..Default::default()
    }

(The `..Default::default()` spread should cover it, but some
literals don't use the spread.)

### `iced::theme::Palette`

New `warning: Color` field. Patch every literal:

    Palette {
        background: Color::WHITE,
        text: Color::BLACK,
        primary: ACCENT_INDIGO,
        success: SUCCESS_GREEN,
        danger: DANGER_RED,
        warning: WARNING_AMBER,    // NEW — required
    }

## 8. Removed API: `Subscription::run_with_id`

iced 0.14 removed `Subscription::run_with_id` (used to dedupe
subscriptions by an opaque ID).

### Replacement

Use `Subscription::run` with a stable function pointer; iced
0.14's subscription system dedupes by type-id of the function,
not by explicit ID.

### Before

    Subscription::run_with_id(WatcherId, move || stream)

### After

    Subscription::run(|| stream)

1 error; identify the call site + the stable function pointer
to switch to.

## 9. Removed APIs

### `scrollable::Id`

No replacement — scrollable instances are identified by their
parent position in the widget tree. Drop the explicit Id.

### `scrollable::scroll_to`

Method moved to `scrollable::scroll_to` as an associated
function call inside an event handler closure. Different shape.

### `text_input::focus`

Use `iced::widget::focus_next` / `focus_previous` /
`Task::focus(text_input::Id::new("name"))` depending on
intent.

### `Application::run_with`

Use the builder's `.run()` after assembling state via the
closure pattern from §2.

### `BootFn` trait

The `&str: BootFn<_, _>` trait bound disappeared. The error
typically surfaces when passing a literal string to a builder
that previously accepted one as a label; check the new builder
method signature.

## 10. Per-crate port plan

Each UX-PRE.b/.c/.d/.e/.f ports one crate. The bundle order
(smallest first, builds confidence):

1. **UX-PRE.f mde-iced-components** (2 errors) —
   `crates/mde-iced-components/src/lib.rs:99` Space::new ×1 +
   `:193` container::Style.snap ×1. Single-file fix.
2. **UX-PRE.e mde-wizard** (19 errors) —
   `crates/mde-wizard/src/main.rs` Application trait + Space +
   keyboard + horizontal_space + closure-sig changes.
3. **UX-PRE.d mde-voice-hud** (20 errors) —
   `crates/mde-voice-hud/src/main.rs` Application trait + the
   same set as wizard.
4. **UX-PRE.c mde-popover** (57 errors) —
   `crates/mde-popover/src/{audio,minimized,expose,snap_assist,window_actions,app_switcher,weather,watermark,icon_mapper,...}.rs`
   each has its own Application impl (one per mde-popover
   subcommand).
5. **UX-PRE.b mde-portal** (108 errors) — LARGEST.
   `crates/mde-portal/src/{main,portal_full_main,workspace,
   marquee,typewriter,app,...}.rs`.

### Atomic bundling (per §0.12)

The workspace must compile at every commit per §0.12. Three
options:

**Option A — squashed bundle:** all 5 ports + the Cargo.toml
deps bump (UX-PRE.a) in one commit. Pros: workspace is green
at every commit. Cons: large diff, longer review.

**Option B — feature-flagged transition:** add a `iced014`
cargo feature to each crate that selects the new API. Keep
the old API code paths under `#[cfg(not(feature = "iced014"))]`
until every crate ports, then drop the cfg. Pros: incremental
commits. Cons: doubled code surface temporarily.

**Option C — worktree-then-merge:** do all 5 ports inside a
git worktree branch + run cargo check between each crate port
+ when green, merge to main as one squashed commit. Pros: same
visible state as Option A but with intermediate checkpoints in
the worktree branch history.

The ship-CH session recommends **Option C** for the actual
port — the probe worktree approach worked; this just extends
it from "probe" to "port".

## 11. Verification gates

Every per-crate port must pass:

1. `cargo check -p <crate> --all-targets` returns 0 errors.
2. `cargo test -p <crate>` passes (or only fails on
   pre-existing flakes flagged in the commit body).
3. `install-helpers/lint-design-tokens.sh` clean on touched
   files (some `Color::from_rgb` calls may need allow-list
   re-anchoring as line numbers shift).
4. `install-helpers/lint-voice.sh` clean.
5. Visual surfaces re-snapshot — UX-23's snapshot baseline
   may need refresh; document any visual delta in the commit
   body.

After all 5 ports + the deps bump merge to main:

6. `cargo build --workspace --all-targets` clean.
7. `make rpm` clean (§0.7.4 gate).
8. `make test-nodeps` clean.

## 12. Risks

| Risk | Mitigation |
|---|---|
| `iced_layershell::Application` builder API differs from the docs example in subtle ways | The 0.18 docs at `application.md` are the canonical reference; test small changes against `cargo check -p <crate>` before committing |
| Subscription dedup behavior changes silently (without explicit `run_with_id`) | Run the mde-portal + mde-popover binaries against the probe and confirm visually that subscriptions don't fan out spuriously |
| Visual snapshot baseline drift | UX-23's snapshot baseline must be updated in the same commit as the port; the gate fails otherwise |
| New Palette.warning field requires a real amber tint, not a placeholder | Use the existing `WARNING_AMBER` from `crates/mde-theme/src/palette.rs` (or define it if missing) |
