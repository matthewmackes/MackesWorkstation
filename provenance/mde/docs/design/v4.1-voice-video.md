## 1. Goal

Bring real-time voice + video + presence + 1:1 chat to MDE, with
seamless reach across the mesh AND out to the PSTN, so an operator
can:

1. **Click any peer in the Workbench and talk / video-call / chat** —
   no manual SIP config, no firewall holes, no third-party SaaS.
2. **Place a call from any peer through any peer when direct
   reach is blocked**, using mesh transit-routing via the relay
   peer's RTPengine (consults mackesd connectivity telemetry).
3. **Dial public phone numbers from any peer** through that peer's
   own Vitelity Communications SIP trunk. Inbound DIDs ring the
   peer they belong to.
4. **Operate a "dual plan"** — one policy bucket controls the
   in-mesh voice topology, a second controls each peer's public
   PSTN edge. Both deploy through the existing Phase-12 draft →
   validated → approved → applied → verified state machine.

**Explicitly out of v4.1.0 (moved to v4.2.0):**
multi-party conferencing (`ConfBridge`-equivalent), voicemail,
music-on-hold, intercom / page, ring groups. The 2026-05-24
operator lock declared these PBX features non-essential to the
first cut; v4.2.0 will pick a media-server stack
(FreeSWITCH / Janus / similar) once Kamailio + RTPengine have
shipped.

## 2. Audience + scope locks

Unchanged from the v12 connectivity-scope survey:

- **Peer cap:** ≤16 (Q2).
- **Geography:** single region (Q4) — sub-100 ms RTT between any
  two peers via the best path.
- **Same-LAN ratio:** ~50% (Q3) — half of peer-pairs share a
  broadcast domain; the other half must reach via relay /
  HTTPS-443 fallback.
- **Headless first-class:** (Q5) — the Pi-class Host peer must
  serve voice the same as a GUI peer. The embedded client is
  optional on a peer; the per-host Kamailio daemon is not.
- **Platform:** Linux desktop only (Q21). Android / iOS softphone
  parity is out of scope.

## 4. Per-host Kamailio daemon (Layer 2)

### 4.1 Distribution

- **Package:** `kamailio` 5.8.x from Fedora 44's official repo.
  Pulled in by `packaging/fedora/mackes-shell.spec`'s `Requires:`.
  Bundled module set includes `tm`, `sl`, `tls`, `dispatcher`,
  `presence`, `presence_xml`, `pua_dialoginfo`, `msilo`, `uac`,
  `uac_redirect`, `rtpengine`, `auth`, `auth_db`, `usrloc`,
  `registrar`, `path`, `htable`, `dialog`, `xlog`.
- **Service unit:** `data/systemd/kamailio-mde.service` —
  separate from any upstream `kamailio.service`. Runs as a
  dedicated `_kamailio_mde` user with its own
  `/var/lib/kamailio-mde/` data root, `/etc/kamailio-mde/`
  config tree, and `/var/run/kamailio-mde/` runtime dir so a
  parallel upstream Kamailio install (if any) keeps its config
  untouched.
- **CLI namespacing:** `mde voice kamcmd <args>` →
  `kamcmd -s /var/run/kamailio-mde/kamcmd.sock <args>` against
  the MDE socket. The bare `kamcmd` continues to target the
  upstream daemon if present.

### 4.2 Binding

Kamailio binds to two interfaces only:

1. `127.0.0.1:5060` UDP — for the embedded PJSIP client. UDP is
   intentional here: same-host loopback, no need for the TLS
   overhead.
2. `nebula1:5061` TLS — for inter-peer SIP signaling. The
   Nebula tun device is up by the time we start
   (`After=nebula.service`); pjsip-style binding to the
   interface IP is enforced via Kamailio's `listen=` directives
   in the generated cfg.

The Vitelity REGISTER session is **outbound only** — Kamailio's
`uac` module opens a TLS connection to `out.vitelity.net:5061`;
Vitelity sends inbound INVITEs back over the established session
(per RFC 5626 / outbound proxy). **No port 5060 is exposed on any
public interface, ever.**

### 4.3 Location & dispatcher

Each peer's Kamailio has exactly **one** stable location entry
(the embedded PJSIP client) plus one entry per **remote** peer:

| Source                | Storage                                  |
|-----------------------|------------------------------------------|
| `mde-local`           | usrloc — populated by REGISTER from PJSIP|
| Remote peer `1NNN`    | `dispatcher.list` — populated by VV-2 from `voice_mesh` policy |
| Vitelity trunk        | `uacreg.list` — populated by VV-2 from this peer's `voice_public` policy |

`dispatcher.list` is the canonical "peer 1NNN → SIP URI"
mapping. Reloaded via `kamcmd dispatcher.reload` whenever
mackesd's policy changes; no Kamailio restart required.

The `1NNN` extension is the 4-digit form of the lexicographic
index of the peer's `node_id`. With a 16-peer cap, extensions
run `1001`–`1016`. Generated; not operator-editable. Operators
attach **display names** via the Workbench; the routing stays
numeric.

## 5. Per-host RTPengine (Layer 3)

### 5.1 Why RTPengine

Kamailio is SIP signaling only — it has no media path. RTP frames
need a relay that:

- Cooperates with Kamailio over an NG-protocol UNIX socket so
  every dialog gets allocated RTP ports the kernel can NAT.
- Terminates SRTP-DTLS on each leg (per design §7.4) so call
  audio is encrypted in transit even over the Nebula tunnel.
- Relays without transcoding (operator-locked 2026-05-24): the
  embedded PJSIP client and Vitelity negotiate codecs end-to-end;
  RTPengine just shuttles frames.

### 5.2 Distribution

- **Package:** `rtpengine` from Fedora 44 (currently
  `ngcp-rtpengine-11.5.x`) via the official repo. No
  third-party COPR.
- **Service unit:** `data/systemd/rtpengine-mde.service`. Runs
  as a dedicated `_rtpengine_mde` user. Listens on the NG
  socket only; RTP traffic uses kernel iptables marks the
  Kamailio cfg sets.
- **Port range:** `30000–40000/udp`, bound to `nebula1` for
  mesh-side flows and `127.0.0.1` for embedded-client-bound
  legs. Same defense-in-depth rule as Kamailio — never on a
  public iface.

### 5.3 No transcoding

Operator-locked 2026-05-24: RTPengine ships without the
transcoding plugin enabled. Consequences:

- **Mesh-to-mesh:** Opus end-to-end (preferred), VP8 video
  end-to-end. RTPengine relays SRTP frames byte-identical at
  the boundary.
- **PSTN out:** the embedded PJSIP client negotiates PCMU with
  Vitelity. The mesh leg from the calling peer's PJSIP client
  is PCMU all the way through Kamailio + RTPengine; Vitelity
  receives PCMU. (Trade-off: lower mesh-side fidelity for PSTN
  calls vs the original Asterisk transcoding plan. Acceptable
  given PSTN is the worst link in the chain anyway.)
- **PSTN in:** Vitelity sends PCMU; the PJSIP client renders
  PCMU. No transcoding boundary.
- **Conference:** unavailable in v4.1.0 — moved to v4.2.0 where
  a media server gets picked.

## 6. Mesh routing (Layer 5, Kamailio cfg)

### 6.1 Direct dial

The generated `kamailio.cfg` has a `route[MESH]` block:

```text
route[MESH] {
    # Dial pattern 1NNN — mesh peer index.
    if ($rU =~ "^1[0-9]{3}$") {
        $var(target) = $sht(mesh=>$rU);
        if ($var(target) == $null) {
            sl_send_reply("404", "peer not in mesh roster");
            exit;
        }
        if (!ds_select_dst("1", "4")) {
            xlog("L_INFO", "mesh: dispatcher returned no destination for $rU\n");
            sl_send_reply("503", "no transit available");
            exit;
        }
        # Hand the call to RTPengine and record-route so the
        # response path comes back through us.
        rtpengine_offer();
        record_route();
        t_on_reply("REPLY_RTP_ANSWER");
        t_relay();
        exit;
    }
}
```

The `dispatcher` module is fed from `dispatcher.list` (generated
by VV-2). When the destination peer's direct path is reachable,
ds_select_dst picks the direct route; when only transit is
viable, it picks the transit peer.

### 6.2 Transit

Kamailio doesn't B2BUA the way Asterisk did. Instead it forwards
with record-route — both endpoints stay visible to each other,
but every SIP message + every RTP frame passes through the
transit peer. The transit peer's RTPengine relays the SRTP
streams; the transit peer's Kamailio doesn't terminate the
dialog.

This is *more transparent* than Asterisk's B2BUA approach (no
double-leg, no CDR doubling) and a strict superset for our
purposes (record-route lets Kamailio see in-dialog messages).

The `mackesd_core::voice::best_path` heuristic still applies —
prefer paths under 80 ms RTT / 5% loss; throughput is irrelevant
to the bounded Opus / VP8 voice load. The dispatcher.list rows
get weighted from this heuristic on every policy refresh.

## 7. Vitelity PSTN edge (Layer 4) — per-peer

### 7.1 Account model

Every peer that wants public-network reach has its own Vitelity
sub-account. From the Vitelity portal:

- Username (per-peer, e.g., `mde-alice-laptop`).
- Password.
- Outbound proxy: `out.vitelity.net:5061` (TLS).
- Inbound DID(s): one or more E.164 numbers Vitelity routes to
  this sub-account.
- Outbound CID: a verified DID owned by the operator (Vitelity
  rejects unverified CID).

These are entered **only** through the Workbench Voice →
Vitelity panel (VV-13). The operator never edits the cfg
directly.

### 7.2 Inbound — `route[VITELITY_IN]`

```text
route[VITELITY_IN] {
    # Match by source — INVITE arrived on the registered
    # uac dialog, so the From-uri is Vitelity's.
    $var(did) = $rU;
    $var(rule) = $sht(vitelity_did=>$var(did));
    switch ($var(rule)) {
        case "ring-self":
            $ru = "sip:mde-local@127.0.0.1:5060";
            break;
        case "ring-extn":
            # voice_public stores the redirect target.
            $ru = $sht(vitelity_did_target=>$var(did));
            break;
        # ring-group / voicemail / confbridge / ivr — all
        # deferred to v4.2.0 with the rest of the PBX feature
        # set.
        default:
            sl_send_reply("404", "no inbound rule for DID");
            exit;
    }
    rtpengine_offer();
    t_relay();
}
```

### 7.3 Outbound — `route[VITELITY_OUT]`

```text
route[VITELITY_OUT] {
    # Dial pattern 9NXXNXXXXXX — US/CA PSTN.
    if ($rU =~ "^9[0-9]{10}$") {
        $var(dst) = $(rU{s.substr,1,0});
        # Negotiate PCMU end-to-end — no transcoding.
        rtpengine_offer("codec-strip-Opus codec-mask-PCMU");
        # Use the per-peer Vitelity registered binding as the
        # outbound socket.
        $du = "sip:" + $var(dst) + "@out.vitelity.net:5061;transport=tls";
        $var(cid) = $sht(voice_public=>"cid");
        uac_replace_from("Anonymous", "sip:" + $var(cid) + "@out.vitelity.net");
        t_relay();
        exit;
    }
}
```

### 7.4 Failure mode

If outbound REGISTER fails (auth, network, outage):

1. Kamailio's `uac` module marks the registration as `failed`.
2. A new mackesd worker (VV-14 sub-task) polls `kamcmd
   uac.reg_dump` every 30 s and surfaces a non-blocking
   diagnostics chip on the status cluster.
3. **No traffic spill.** Calls to `9NXXNXXXXXX` from this peer
   fail with `503 Vitelity unreachable`. Mesh calls continue
   unaffected (separate route block).
4. Retry REGISTER with the same gentle exponential backoff the
   connectivity layer uses.

## 8. Embedded SIP/video client (Layer 6)

Unchanged from the 2026-05-23 PJSIP-via-Rust-FFI lock:

- `crates/mde-voice-pjsip-sys` — `bindgen` against system
  `pjproject-devel`. **Note:** PJSIP is the C SIP UA stack; the
  *server* swap from Asterisk to Kamailio doesn't affect it.
  The embedded client speaks SIP to its local Kamailio just as
  cleanly as it would have to a local Asterisk.
- `crates/mde-voice-client` — safe Rust wrapper, async over
  tokio, exposes `Call`, `Registration`, `Presence`,
  `MessageSession`, `MediaSink`, `MediaSource`.
- Audio capture / playback via PipeWire.
- Video capture via XDG `org.freedesktop.portal.Camera`.
- Codecs:

| Direction        | Codec set (priority order)                  |
|------------------|---------------------------------------------|
| Audio (mesh)     | Opus @ 24 kbps                              |
| Audio (PSTN)     | PCMU @ 64 kbps (Vitelity's preferred)       |
| Video (mesh)     | VP8 @ 600 kbps                              |

No video over PSTN (Vitelity trunks are audio).

## 9. Presence + chat (Layer 4 features)

### 9.1 Presence

Kamailio's `presence` + `presence_xml` modules host SIMPLE
presence (RFC 3856). Every embedded PJSIP client publishes its
state via SIP PUBLISH; every other peer's PJSIP subscribes via
SIP SUBSCRIBE. Kamailio aggregates and notifies.

| State      | Trigger                                                |
|------------|--------------------------------------------------------|
| `available`| Embedded client running, no active call                |
| `on-call`  | At least one INVITE in a `Confirmed` dialog            |
| `away`     | Operator idle >5 min (reuses sway-idle timer)          |
| `dnd`      | Operator toggled DND in the slide-up modal             |
| `offline`  | Kamailio's usrloc shows the AOR as `expired`           |

Subscriptions: every peer subscribes to every other peer's AOR.
16 × 15 = 240 subscriptions across the mesh — trivial for
Kamailio (it routinely handles thousands).

### 9.2 Text chat (SIP MESSAGE)

Kamailio's `msilo` (Message Silo) module — stores MESSAGE
requests for offline peers and delivers on next REGISTER.
Online delivery is a straight forward through the dialog plane.

The embedded client persists history locally in
`~/.local/share/mde/voice/chat.sqlite` (per-device by design).

(Group chat moves to v4.2.0 alongside ConfBridge — Kamailio has
no native group-message broadcast that doesn't require a back-
end like Matrix or a media server.)

## 10. The "dual plan" — policy buckets

(Unchanged from the original lock — Kamailio swap doesn't
change the policy shape.)

Per the existing Phase-12 state model, Voice adds **two new
policy kinds** in `crates/mackesd/src/policy/`:

### 10.1 `voice_mesh` policy

What it controls:

- Extension-to-peer assignment (default: lexicographic
  `node_id` → `1NNN`; operator can override).
- Display names.
- Transit-routing preferences (which peers may serve as
  transit relays — defaults to all GUI peers, headless peers
  off by default to keep their CPU free).
- Presence subscription policy (full mesh by default;
  operator can restrict to named groups).

### 10.2 `voice_public` policy (per peer)

What it controls **on this peer**:

- Vitelity sub-account credentials (username, password,
  outbound proxy URL).
- Owned DID list + per-DID inbound rule (`ring-self`,
  `ring-extn`; `ring-group` / `voicemail` / `confbridge` / `ivr`
  deferred to v4.2.0).
- Outbound rules (digit-pattern matching, CID).
- Outbound permission policy.

Both policy kinds land through the existing draft → validated →
approved → deploying → applied → verified lifecycle.

## 11. Configuration generation flow (Layer 1)

```text
desired_config (voice_mesh + voice_public + nodes)
        │
        ▼
mde-voice-config::generate()        ← pure function
        │
        ├──→ /etc/kamailio-mde/kamailio.cfg
        ├──→ /etc/kamailio-mde/dispatcher.list
        ├──→ /etc/kamailio-mde/uacreg.list
        └──→ /etc/rtpengine-mde/rtpengine.conf
        │
        ▼
kamailio-mde reload                  ← graceful, via kamcmd
rtpengine-mde reload                 ← SIGHUP, picks up new
                                       interface bindings only
        │
        ▼
observed_telemetry: REGISTRATION events
        │
        ▼
mackesd reconciler: Applied → Verified
```

The generator is a pure function (input → output, no I/O). Tests
use golden fixtures: a `desired_config` JSON in
`crates/mde-voice-config/tests/fixtures/` produces a known set
of generated files asserted via `insta` snapshot diffs.

## 13. Acceptance / Definition of Done

A v4.1.0 cut is `[✓]` when:

1. **Two-peer mesh call** — operator at peer A dials `1002`
   from the embedded client; peer B's embedded client rings;
   bidirectional Opus audio + VP8 video for 60 s with no
   gap > 100 ms. RTPengine on both ends relays SRTP frames.
2. **Three-peer transit call** — peer A's direct path to peer
   C is artificially blocked (iptables drop on `nebula1`
   between A and C); A dials `1003`; B's Kamailio record-routes
   the call; B's RTPengine relays media; audio quality on par
   with direct.
3. **PSTN outbound** — peer A dials `915551234567`; Vitelity
   places the call (PCMU end-to-end through A's RTPengine);
   the called PSTN endpoint shows A's CID.
4. **PSTN inbound** — caller dials A's owned DID; peer A's
   embedded client rings within 3 s; audio is bidirectional
   PCMU.
5. **Presence** — peer B goes idle; peer A's PJSIP client (via
   Kamailio NOTIFY) shows B as `away` within 30 s.
6. **Chat** — peer A sends MESSAGE to peer B; B's client
   surfaces it within 1 s; history persists across restart.
   Offline delivery via `msilo` works (B is offline; A sends;
   B comes online; MESSAGE delivers).
7. **Vitelity outage drill** — `iptables -A OUTPUT -d
   out.vitelity.net -j DROP` on peer A; peer A's
   `voice_public` status flips to `Unavailable` within 60 s
   without spilling traffic; mesh calls continue uninterrupted.
8. **Dual-plan deploy** — operator submits a `voice_public`
   revision (new DID rule) without touching `voice_mesh`;
   only peer A's Kamailio reloads `uacreg.list`; no other
   peer's cfg rewrites.

(Removed: 4-peer ConfBridge drill — moves to the v4.2.0 epic
with the ConfBridge / voicemail re-introduction.)

## 14. Risks + open decisions

| Risk                                    | Mitigation                                  |
|-----------------------------------------|---------------------------------------------|
| Kamailio cfg DSL learning curve         | Generator owns it; operator never writes    |
|                                         | cfg by hand. The cfg is reviewable in PRs.  |
| Kamailio 5.8 vs F45's 5.9               | Track upgrade in a separate worklist item   |
|                                         | when F45 lands.                             |
| RTPengine kernel module                 | We stay user-space (no kernel module)       |
|                                         | until we have a HW perf bench — see VV-15.  |
| Vitelity API rate-limit                 | Cache verified-DID list per 5-min TTL;      |
|                                         | only refresh on Workbench panel open.       |
| Camera portal under sway/wlroots        | Fall back to v4l2 if portal unavailable;    |
|                                         | log + non-blocking diagnostics chip.        |
| First-call latency vs Q8 (≤3 s)         | Pre-register at boot; INVITE adds ~200 ms   |
|                                         | for SRTP-DTLS + ICE.                        |
| PSTN call quality (no transcoding)      | PCMU end-to-end means the embedded PJSIP    |
|                                         | client must speak PCMU on its outbound      |
|                                         | offer. The operator-locked trade-off.       |

Open decisions to revisit at code-time:

- **CDR storage** — `acc` text file vs SQLite. Default: text
  file (rotated daily). v4.2 may revisit.
- **E.911 / emergency-call handling** — Vitelity supports E911
  registration; the Vitelity panel should require an address
  before activating the trunk. Out of v4.1 unless legal review
  surfaces it sooner.
- **TLS cert provisioning** — Kamailio's TLS cert is the same
  mesh-issued cert mackesd already manages at
  `~/.config/mde/certs/mesh-host.pem`. VV-1 ships a symlink or
  copy; the TLS-rotation drill happens with the next cert
  rotation.
