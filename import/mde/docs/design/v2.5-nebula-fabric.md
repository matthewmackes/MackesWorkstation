**Status:** authoritative; supersedes the Tailscale/Headscale/DERP
locks in `v12.0-enterprise-mesh.md` and `v12-connectivity-scope.md`
for everything below the `mackesd_core` library facade (Layer 7).

**Decision driver:** the v12 stack — Headscale coordinator +
Tailscale data plane + self-hosted `derper` relay + an
`Https443` parallel fallback — accreted four discrete daemons,
two coordination protocols, and a SaaS dependency on Tailscale's
public DERP map. The platform locks (≤16 peers, single region,
self-hosted-first, sub-3 s first packet) do not justify that
surface area. Nebula collapses control plane + data plane +
relay rendezvous into one fabric whose lighthouse pattern maps
1:1 onto `mackesd`'s existing leader election. With no users
yet on a Tailscale-built mesh (greenfield lock — Q5 of the
2026-05-23 survey), the cost of the swap is bounded to
implementation effort; no migration code is owed.

## 5-question survey — locks

| # | Area          | Question                                                       | **Lock** |
|---|---------------|----------------------------------------------------------------|----------|
| 1 | Scope         | How aggressive should the rip-and-replace be?                  | **Full replacement.** Headscale + Tailscale + `derper` removed entirely. Nebula is the only mesh fabric. `mackes-transport`'s `DerpRelay` variant deleted; `DirectUdp` is repurposed as Nebula's tun. Wizard, `mesh_services.py`, `mesh_vpn.py` rewritten or removed. |
| 2 | Lighthouse    | Which peers run a Nebula lighthouse?                           | **Every Host-role peer.** Reuses the existing `leader.rs` election + failover heartbeat. 1–3 lighthouses typical at ≤16 peers. A non-Host peer never serves rendezvous traffic. |
| 3 | PKI / CA      | Where does the Nebula CA private key live?                     | **Dedicated CA on the leader.** `mackesd` mints a fresh Nebula CA at mesh-init, seals the private key under `/var/lib/mackesd/nebula-ca/` on the leader (mode 0600, root-owned). `mackesd enroll` signs peer certs server-side. Failover = epoch bump + re-issue all peer certs from the new leader. CA private key never leaves the leader. |
| 4 | Q10 covert    | What happens to the TCP/443 covert transport?                  | **Tunnel Nebula over TCP/443.** A new `mackes-nebula-https-tunnel` crate wraps Nebula's UDP frames in a rustls TLS 1.3 stream over TCP/443, byte-indistinguishable from a curl-to-nginx baseline. No parallel transport; one fabric end-to-end. |
| 5 | Migration     | How do existing 1.x meshes migrate?                            | **Greenfield only.** There are no users on a Tailscale-built mesh. v2.5 is the first release that ships a working fabric. No migration code, no compat shims, no dual-stack window. |

These locks supersede:

- `v12-connectivity-scope.md` Q6 ("self-hosted DERP first,
  public Tailscale DERP fallback") — **retired.** No DERP at
  all; Nebula's lighthouse pattern serves both rendezvous and
  relay.
- `v12-connectivity-scope.md` Q10 ("TCP/443 only — make it
  indistinguishable from real HTTPS") — **kept, implementation
  changed.** Covert transport now wraps Nebula frames, not a
  separate WireGuard-replacement.
- `v12.0-enterprise-mesh.md` Layer 0 (process supervisor)
  references to `tailscaled` / `headscale` / `derper` —
  **retired.** Replaced by a single `nebula.service` per peer +
  conditional lighthouse activation.

## CA / cert lifecycle

```text
mesh-init (operator runs `mackesd mesh init`)
  │
  ▼
mackesd::ca::mint_ca()
  ├─ generate Nebula CA keypair (Curve25519)
  ├─ self-sign CA cert with mesh-id + epoch=0
  ├─ store private key sealed at /var/lib/mackesd/nebula-ca/ca.key
  │   (mode 0600, root:root)
  └─ store public CA cert at /var/lib/mackesd/nebula-ca/ca.crt
       (mode 0644 — replicates to every joining peer)

mackesd enroll  (joining peer's request lands in the leader's inbox)
  │
  ▼
mackesd::ca::sign_peer_cert(node_id, public_key, role)
  ├─ verify Ed25519-signed EnrollmentRequest (existing mackesd_core::identity)
  ├─ allocate Nebula overlay IP from the mesh CIDR (10.42.0.0/16 default)
  ├─ build peer-cert (groups=[role:host|peer], name=<node_id>)
  ├─ sign with CA private key
  └─ write the signed cert + mesh CIDR + lighthouse roster
     into ~/QNM-Shared/<peer>/mackesd/nebula-bundle.json

joining peer reads the bundle via its watcher, materializes
/etc/nebula/{ca.crt,host.crt,host.key,config.yaml}, and starts
nebula.service. If the peer's role is Host, the
ConditionPathExists guard on nebula-lighthouse.service also fires.

leader failover (existing leader.rs lease expires)
  ├─ new leader detects expired lease via fs2 advisory lock
  ├─ mackesd::ca::rotate_ca_on_promotion()
  │   ├─ bump CA epoch counter in nebula_ca SQL table
  │   ├─ re-mint CA keypair under the new epoch
  │   └─ re-sign every existing peer cert with the new CA
  └─ pushes new bundle JSON to every peer's QNM-Shared inbox
     (peers reload nebula.service when nebula-bundle.json changes)
```

Mesh CIDR default `10.42.0.0/16` is wide enough that the 16-peer
hard cap (Q-MX18) never collides with the address space.

## Lighthouse topology — Q2 follow-through

Every Host-role peer ships a `nebula-lighthouse.service` unit
gated on `ConditionPathExists=/var/lib/mackesd/nebula/role.host`.
`mackesd` writes that marker file when the leader-election lease
or a manual `mackesd promote <node>` call promotes the node to
Host. Demotion (lease expiry without renewal, or `mackesd demote
<node>`) removes the marker; systemd's `Restart=on-failure` +
`Conflicts=` semantics ensure the lighthouse process exits within
~5 s of demotion.

At any given moment the mesh has between 1 and 3 lighthouses
(typical operational target). The roster — the lighthouse static
IP list every peer's `nebula.yaml` pins under `lighthouse.hosts`
— is computed by `mackesd_core::topology::lighthouse_roster()`
and written into the per-peer bundle JSON. Roster changes (Host
demotion, new Host promotion) trigger a fresh bundle write to
every peer.

**Failure modes covered:**

1. **Single lighthouse alive** — rendezvous still works; new
   peers can join.
2. **All lighthouses unreachable simultaneously** — existing peer
   pairs stay connected via cached host-map entries (Nebula's
   `static_host_map`). New peers can't rendezvous until at least
   one lighthouse returns. Surfaced in the panel as a
   degraded-mesh banner per Q11.
3. **Leader-election lease expires while peer is mid-handshake** —
   handshake completes against the old lighthouse cert (CA epoch
   N); peer re-enrolls within ~10 s under the new epoch.

## TCP/443 covert transport — Q4 follow-through

New crate: `crates/mackes-nebula-https-tunnel/`.

**Wire format:** every Nebula UDP frame is wrapped in a 4-byte
length-prefixed payload over a single long-lived rustls TLS 1.3
stream. The TLS server cert is the same Let's Encrypt cert the
lighthouse serves (alpn=`h2,http/1.1`). To a passive observer the
stream is byte-indistinguishable from a long-poll HTTP/2 session.

**Activation policy** (port from the retired
`https_fallback.rs::HttpsFallbackState`):

```
state machine:  Inactive → Activating → Active → Failing → Inactive
trigger:        3 consecutive Nebula direct-UDP + lighthouse-relay
                failures within a 30 s window
```

**Server side:** the lighthouse(s) accept both `:4242/udp`
(native Nebula) and `:443/tcp` (TLS-wrapped Nebula). Frame
demuxing happens before the Nebula crypto layer sees the packet,
so the inner Nebula stack is unmodified.

**Throughput target:** the Q16 ≥900 Mbps LAN target does NOT
apply to the TCP/443 path. The covert path's target is "any
forward progress on a network where UDP is blocked." Realistic
ceiling ~50 Mbps single-stream over modern TLS 1.3, more than
enough for control-plane traffic + interactive shells. Bulk file
sync (mesh-FS / `mesh_media.py`) gates on UDP availability and
degrades to "queued until UDP returns" when only the TCP path is
live — surfaced in the panel via the existing
`HealthDegraded(Https443)` SwitchReason.

## What the connectivity SLOs become

| Lock                                  | v12 target          | v2.5 target          | Why it changes                                       |
|---------------------------------------|---------------------|----------------------|------------------------------------------------------|
| First-packet latency (Q8)             | < 3 s               | < 1 s                | Nebula's lighthouse rendezvous + UDP hole-punching beats Headscale's NetInfo handshake. |
| LAN-direct detection (Q7)             | < 30 s              | < 10 s               | mDNS still drives detection but `static_host_map` injection fast-paths the data plane. |
| LAN-direct throughput (Q16)           | ~900 Mbps           | ~900 Mbps            | Nebula uses ChaCha20-Poly1305 in userspace; same envelope as Tailscale's userspace WireGuard. |
| Covert TCP/443 throughput             | ≥ 5 Mbps (Q10)      | ≥ 5 Mbps             | Unchanged — covert path is for control + interactive only. |
| Roaming handoff (Q22)                 | < 10 s              | < 5 s                | Nebula handshakes are stateless on the relay side; reconnect window halves. |

## What this design rules OUT

- **No multiple control planes.** Headscale + Tailscale OAuth are
  gone. `mackesd_core::ca` is the only enrollment authority.
- **No SaaS dependency.** No call-out to Tailscale's coordination
  server, no public DERP map. The mesh runs without an internet
  uplink (modulo the Let's Encrypt cert renewal for the TCP/443
  lighthouse listener — fail-closed: existing cert serves up to
  90 days past renewal failure).
- **No userspace WireGuard.** The platform stops carrying WireGuard
  as a dependency entirely.
- **No external IdP.** PKI is internal to the mesh. SSO / Keycloak /
  Auth0 integration was never in scope (Q1+Q5 of v12 survey) and
  Nebula's PKI-only model commits to that lock permanently.

## Acceptance criteria for v2.5 cut

1. **`mackesd mesh init` on a fresh host** mints CA, brings up
   lighthouse, allocates overlay IP, prints the join token.
2. **`mackesd enroll` on a second host** with the join token from
   step 1 → both hosts ping each other on the Nebula overlay
   within 30 s.
3. **Power-off + replug LAN cable** on either host → reconnect
   under 5 s, verified by the `LinkWatchWorker` + the panel's
   transition indicator.
4. **Block UDP egress on host A's firewall** → traffic
   transparently fails over to the TCP/443 path within the
   `HttpsFallbackState`'s 30 s probe window; panel shows the
   `HealthDegraded(NebulaHttps443)` SwitchReason.
5. **Promote a third host to Host role** → it appears in every
   peer's `lighthouse.hosts` list within 10 s; demoting it
   removes the marker file and the unit stops within 5 s.
6. **Kill the leader** → election lease expires, a new Host wins
   the lease, CA epoch bumps, every peer gets a fresh cert
   bundle, mesh continues operating with no operator action.

All six are user-felt SLOs — bench-validated on the 6-peer test
fleet, 7-day window, per the Q25 acceptance lock.
