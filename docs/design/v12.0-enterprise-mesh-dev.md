# Phase 12 — Enterprise Mesh developer guide

Pair with `v12.0-enterprise-mesh.md` (architecture) and
`docs/help/mesh-ops.md` (operator runbook). This doc explains how
to extend the backend — adding policy kinds, dispatching from the
reconciler, computing topology diffs.

## Adding a new policy kind

Policies are JSON documents validated against a known schema
(12.7.2). Every kind has three pieces:

1. **A `Policy` enum variant** in `crates/mackesd/src/policy/types.rs`.
2. **A JSON schema** under `crates/mackesd/schemas/policy/<kind>.json`.
3. **A reconciler dispatcher** in
   `crates/mackesd/src/reconcile/policy_dispatch.rs` that knows
   what runtime action to take when this policy applies.

### Example — a hypothetical `allow_east_west` policy

```rust
// crates/mackesd/src/policy/types.rs
pub enum Policy {
    AllowEastWest {
        from_region: String,
        to_region: String,
        bandwidth_cap_mbps: Option<u32>,
    },
    // existing variants …
}
```

```json
// crates/mackesd/schemas/policy/allow_east_west.json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "type": "object",
  "required": ["from_region", "to_region"],
  "properties": {
    "from_region": { "type": "string" },
    "to_region":   { "type": "string" },
    "bandwidth_cap_mbps": { "type": "integer", "minimum": 1 }
  },
  "additionalProperties": false
}
```

```rust
// crates/mackesd/src/reconcile/policy_dispatch.rs
fn dispatch(policy: &Policy, ctx: &mut ReconcileContext) -> Result<()> {
    match policy {
        Policy::AllowEastWest { from_region, to_region, bandwidth_cap_mbps } => {
            ctx.topology.add_allowed_edge(from_region, to_region);
            if let Some(cap) = bandwidth_cap_mbps {
                ctx.topology.set_edge_qos_cap(from_region, to_region, *cap);
            }
            Ok(())
        }
        // existing variants …
    }
}
```

### Adding tests for a new policy kind

Every new kind needs three test files:

- `tests/policy/<kind>_schema.rs` — golden-fixture validation tests
  (well-formed accepts, malformed rejects).
- `tests/policy/<kind>_dispatch.rs` — pure-function tests of the
  reconciler dispatcher against a synthetic `ReconcileContext`.
- `tests/policy/<kind>_conflict.rs` — assertions that the new
  kind correctly raises `PolicyConflict` when it contradicts an
  existing rule (per 12.7.2).

## How the reconciler dispatches

The reconcile loop runs every 30 seconds (configurable). Each
tick:

1. **Snapshot.** Reads `desired_config`, `runtime_state`, and
   `observed_telemetry` into immutable views.
2. **Calculate.** Runs `topology::calculate(&desired, &policies)`
   to produce the expected peer adjacencies + route tables. This
   is a **pure function** — no I/O, fully deterministic, tested
   with golden fixtures.
3. **Diff.** Compares `calculated_topology` to `observed_telemetry`.
   Produces `Vec<DriftRow>` with `severity` ∈ { `auto-repairable`,
   `manual-review` }.
4. **Apply.** For `auto-repairable` drift whose policy allows,
   the dispatcher walks the policies in declaration order and
   calls `policy_dispatch::dispatch(policy, &mut ctx)` for each.
   The context accumulates the side effects (Tailscale ACL writes,
   Headscale route updates, mesh-FS reconfigures); a single
   transaction at the end commits them.
5. **Manual review.** Remaining drift surfaces in the Pending
   Changes inbox (`pending_changes` table) for operator approval.

The dispatcher never makes network calls directly — it accumulates
intent into `ReconcileContext` and lets the deploy layer (layer 1
in the architecture diagram) execute. This separation makes the
dispatcher pure-testable + lets the dry-run path (`mackesd apply
--dry-run`, per 12.7.4) run the full validation + plan flow
without touching the network.

## How the topology diff is computed

The topology diff has three modes (per 12.9.3):

- **Desired** — render `calculated_topology` only.
- **Actual** — render observed peer adjacencies from
  `observed_telemetry`.
- **Diff** — render both, overlay differences.

The diff overlay is a set-difference of edges:

```text
diff.missing   = desired.edges \ actual.edges   # red — should-exist
diff.extra     = actual.edges \ desired.edges   # amber — exists-but-shouldn't
diff.healthy   = desired.edges ∩ actual.edges   # rendered normally
```

Edge equality is `(node_a, node_b, kind)` where `kind` distinguishes
direct UDP from DERP-relayed from HTTPS-tunneled (per the
connectivity items in Phase 12.14–12.23). Two edges between the
same pair of nodes via different transports diff independently
— the operator can see "the direct path is up but the relay
path is also up, that's fine."

### Where the diff is rendered

The Cairo renderer (12.9.1) consumes the `TopologyDiff` value
directly. Nodes are positioned by force-directed layout
(deterministic given the same input — for snapshot testing per
12.11.4) and edges are drawn in three passes:

1. **Healthy edges** first (solid lines, normal palette).
2. **Drift `extra`** edges second (amber dashed lines).
3. **Drift `missing`** edges last (red dashed lines on top, so
   they win the z-order).

Labels and hit-test regions are computed after all edges so the
click-target areas don't include drift lines accidentally.

## Hash chain implementation

The audit chain (12.6.3 + 12.10.3) uses SHA-256 over a fixed-shape
payload:

```rust
fn next_hash(prev_hash: &[u8; 32], payload: &EventPayload, ts: i64) -> [u8; 32] {
    let mut hasher = Sha256::new();
    hasher.update(prev_hash);
    hasher.update(serde_json::to_vec(payload).unwrap());
    hasher.update(ts.to_le_bytes());
    hasher.finalize().into()
}
```

`mackesd audit verify` walks the table forward, recomputes each
row's hash, and reports the first mismatch. A successful verify
output looks like:

```text
verified 14392 events  ·  chain intact  ·  last hash 4f3a…b21c
```

A failure looks like:

```text
verified 8744 events  ·  break at event_id 8745 (revision r-2026-05-19-0042)
expected prev_hash 5a2f…  got 9c1e…
```

Treat the break point as your last trustworthy moment. Anything
after needs manual review against the store backups (per
`mesh-ops.md` "Reading the audit log").
