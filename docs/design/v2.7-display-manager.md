# v2.7 — Display manager swap (LightDM → greetd + regreet)

**Locked:** 2026-05-24 via 10-question display-manager survey.
**Worklist:** `docs/PROJECT_WORKLIST.md` § DM-1..DM-8.
**Operator help:** `docs/help/display-manager.md`.

## The 10 locks (verbatim from the survey)

The DM-* epic implements these locks. Each is preserved here as
the canonical design-doc copy so a future audit can verify the
shipped behavior against the lock without spelunking the worklist
preamble.

### Q1 — Choose the greeter substrate

**Lock:** `greetd` + `regreet` running under `cage`.

`greetd` is the minimal display manager that owns the systemd
unit + sessions concept; `regreet` is the GTK4-based greeter UI;
`cage` is the single-surface Wayland compositor that hosts regreet
before login. The trio is the canonical Wayland-native DM stack
on Fedora.

### Q2 — Wayland-only

**Lock:** `mde.desktop` is the only session entry; LightDM's
multi-session X11/Wayland picker is retired.

Multi-environment systems can still install other DMs alongside;
MDE's greeter intentionally shows MDE only.

### Q3 — Username field behavior

**Lock:** type the username every time. No `remember_user`.

A shared workgroup peer shouldn't surface the last user that
logged in. This is a small ergonomic cost for a meaningful privacy
gain.

### Q4 — Auto-login

**Lock:** off by default. No `[initial_session]` block in the
shipped `greetd` config.

If a single-user peer wants auto-login, the operator wires it
manually post-install. MDE itself doesn't ship it.

### Q5 — Session memory

**Lock:** `remember_session = false`. Pick the session every
login.

In practice there's only one session (MDE), so the picker is
mostly cosmetic — but keeping it unmemoised aligns with Q3's
privacy stance.

### Q6 — Power controls

**Lock:** shutdown + reboot + suspend visible in the greeter.
Bottom-right corner, three glyphs.

Greeter-visible power controls let an operator recover from a
locked-out peer without dropping to a TTY.

### Q7 — Visual identity

**Lock:** charcoal background (`#1d1d1f`) + Roboto type + indigo
(`#5b6af5`) accents. Matches the Classic ChromeOS palette the
desktop uses (see `project_chromeos_classic_visual_lock`).

The boot-theater → greeter → desktop sequence should read as one
continuous environment, not three jarring themes.

### Q8 — Header text

**Lock:** `greeting_msg = "Mackes Desktop Environment"`.

Specific over generic — the user is told exactly what they're
about to log into.

### Q9 — Mesh-status indicator

**Lock:** small chip upper-right showing online / joining /
offline. Informational only; doesn't gate auth.

Tracked as DM-7 (Tier 2) — ships in a follow-up commit after
DM-1..DM-6 land. The chip reads from the same overlay-IP publish
file `nebula_supervisor` writes for the rest of the platform.

### Q10 — Rollback plan

**Lock:** LightDM stays installed; rollback is two `systemctl`
commands.

The DM-5 birthright step (`apply_display_manager`) is idempotent
+ doesn't remove the lightdm RPM. The operator help page documents
the rollback procedure.

## Why not keep LightDM?

LightDM is X11-rooted. Its Wayland support is the
`lightdm-wayland-greeter`, which is third-party + lightly maintained.
MDE is Wayland-only post-v2.0 (sway is the locked compositor), so
running an X11-rooted DM on a Wayland-only desktop is a continuous
source of session-leak bugs (XWayland surfaces in the greeter,
mismatched DPI scaling on multi-monitor, etc.).

greetd is the minimal Wayland-native DM. The swap removes a layer
of X11 dependency from the boot path and aligns the greeter with
the rest of the platform.

## Why MDE-only sessions?

A multi-environment system (e.g., GNOME + KDE + MDE on the same
box) typically wants the DM to surface all installed environments.
MDE's greeter intentionally doesn't, for two reasons:

1. **Workgroup peers are MDE-only.** The "MackesDE for Workgroups"
   product story sells a single environment; surfacing alternative
   sessions undermines that.
2. **Stray `*.desktop` files break session memory.** If regreet
   shows three sessions and the user picks one by accident, the
   next login defaults to that — unless Q5's `remember_session =
   false` overrides it. Hiding the alternatives makes the wrong-
   pick failure mode impossible.

If a user genuinely wants to multi-boot environments, they can
add entries to `/var/lib/mde/wayland-sessions/` manually + accept
that they're outside the locked design.
