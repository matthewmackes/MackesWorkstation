# v5.0.0 Compute — KVM + Podman mesh-native virtualization

**Locked:** 2026-05-30 via 10-Q `/plan` survey (VIRT-1..12).
**Status:** v5.0.0 core (§11.1 C10) — operator-elevated.
**Worklist:** `VIRT-1..VIRT-12` in `docs/PROJECT_WORKLIST.md`.

---

## 1. Design locks

| # | Lock | Decision |
|---|---|---|
| L1 | Platform | Libvirt + Cockpit-machines + custom mackesd Bus workers. No OpenNebula, no oVirt, no Nomad. |
| L2 | VM network identity | Each VM gets its own Nebula certificate + IP. First-class mesh peer. |
| L3 | VM Nebula subnet | `10.42.128.0/17` for VMs; peer subnet stays `10.42.0.0/17`. |
| L4 | Service exposure | Explicit per-network selection per port — operator picks mesh / LAN / WAN. No auto-expose. |
| L5 | MeshFS in VM | virtiofsd host-share: host mounts mesh-storage, shares into guest via virtio-fs. |
| L6 | VM disk | Local qcow2 in `/var/lib/mde-vms/`, libvirt storage pool `mde-vms`. |
| L7 | Podman | Unified inventory — KVM VMs + Podman containers share one Bus topic + Workbench panel. |
| L8 | Migration | Cold only: stop VM → rsync qcow2 over Nebula overlay → restart on target peer. |
| L9 | Compute nodes | Always-on: every MDE peer is a compute node. libvirtd socket-activated. |
| L10 | UI surface | New `Compute` panel in `mde-workbench` alongside Mesh, Files, Settings. |
| L11 | CA key | CA key never leaves the CA peer. VM certs signed on request via Bus topic `compute/cert-sign-request/<ulid>` → `cert_authority` mackesd worker replies on `reply/<ulid>`. |

## 2. Architecture

```
┌──────────────────────────────────── Peer A (compute node) ──────────────────────┐
│                                                                                   │
│  mde-workbench (Compute panel)                                                    │
│      └── reads compute/inventory/* (all peers, via Bus subscriber)               │
│      └── sends compute/create, compute/expose, compute/migrate via Bus           │
│                                                                                   │
│  mackesd workers:                                                                 │
│    compute_registry   ── polls libvirtd + podman ── publishes compute/inventory/A │
│    compute_provision  ── virt-install + cert_sign_request + virtiofsd wiring      │
│    compute_expose     ── firewalld rich rules from compute/expose/A               │
│    compute_migrate    ── cold: virsh shutdown → rsync qcow2 → Bus migrate-ready   │
│    cert_authority     ── signs VM Nebula certs (CA peer only, refuses on others)  │
│                                                                                   │
│  KVM Guest (VM A)                                                                 │
│    ├── Nebula cert: 10.42.128.1/17 (signed by CA peer via Bus)                   │
│    ├── virtio-fs mount ← virtiofsd ← /mnt/mesh-storage (host MeshFS)             │
│    └── disk: /var/lib/mde-vms/<vm-id>.qcow2                                      │
│                                                                                   │
│  Podman container (container A)                                                   │
│    └── reported in compute/inventory/A as type=podman                            │
└───────────────────────────────────────────────────────────────────────────────────┘
         │ Nebula overlay 10.42.0.0/16
         │   peers: 10.42.0.0/17
         │   VMs:   10.42.128.0/17
┌────────┴─────────── Peer B ──────────────────────┐
│  KVM Guest (VM B): 10.42.128.2                    │
│  compute_registry publishes compute/inventory/B   │
│  cert_authority (only if B is the CA peer)        │
└───────────────────────────────────────────────────┘
```

## 4. VM Nebula subnet allocation

- **Peer subnet:** `10.42.0.0/17` (unchanged, existing enrollment).
- **VM subnet:** `10.42.128.0/17` (new; `128` is the "VM" bit in the third octet).
- All peer nebula.yaml configs need `unsafe_routes` or a lighthouse route entry for `10.42.128.0/17` — added by `compute_provision` at first VM creation and replicated via mesh-storage.
- The CA peer's `ca.crt` already covers `10.42.0.0/16` (the whole /16); no CA re-issue required as long as the CA was issued for the full /16.

## 5. libvirtd socket activation

To keep the always-on compute model lightweight on low-RAM peers:

```ini
# /usr/lib/systemd/system/libvirtd.socket  (already ships with libvirt)
[Socket]
ListenStream=/var/run/libvirt/libvirt-sock
```

`libvirtd.service` starts on first socket connection. mackesd's `compute_registry` worker connects at startup — so libvirtd is running whenever mackesd is running, which is always. No additional changes needed.

## 6. virtiofsd MeshFS share

For each VM that has `share_meshfs: true`, `compute_provision` adds to the VM's libvirt XML:

```xml
<filesystem type="mount" accessmode="passthrough">
  <driver type="virtiofs"/>
  <source dir="/mnt/mesh-storage"/>
  <target dir="mesh-storage"/>
</filesystem>
```

A `virtiofsd@.service` instance unit is started per VM by `compute_provision`. The guest OS mounts it:

```
mesh-storage /mnt/mesh-storage virtiofs defaults,_netdev 0 0
```

**Dependency:** The host must have mesh-storage mounted. `compute_registry` reports `meshfs_available: bool` in the inventory; the Workbench panel shows a warning badge on VMs with `share_meshfs=true` when `meshfs_available=false`.

## 7. Service exposure — firewalld rich rules

For each `compute/expose` request, `compute_expose` writes a firewalld rich rule per selected network:

```bash
# mesh: forward from Nebula interface (nebula0) to VM's Nebula IP
firewall-cmd --zone=trusted --add-rich-rule='rule family="ipv4" \
  destination address="<nebula0-ip>" port port="<host-port>" protocol="tcp" \
  forward-port port="<host-port>" protocol="tcp" to-addr="<vm-nebula-ip>" to-port="<guest-port>"'

# lan: forward from LAN interface
firewall-cmd --zone=public --add-rich-rule='rule family="ipv4" \
  port port="<host-port>" protocol="tcp" \
  forward-port port="<host-port>" protocol="tcp" to-addr="<vm-nebula-ip>" to-port="<guest-port>"'

# wan: forward from WAN interface (if distinct from LAN)
# — same pattern targeting the WAN zone
```

Rules are persisted (`--permanent` + `firewall-cmd --reload`). `compute_expose` also maintains a `compute/exposed/<peer>` Bus topic listing all active forwarding rules so the Workbench can display them.

## 8. Cold migration flow

1. Workbench Compute panel sends `compute/migrate/<vm-id>` with source + target.
2. Source peer's `compute_migrate` worker: `virsh shutdown <vm>` → polls `virsh domstate` for SHUTOFF (timeout 120 s).
3. `rsync --compress --progress <disk-path> <target-nebula-ip>:/var/lib/mde-vms/` over the Nebula overlay.
4. Publishes `compute/migrate-ready/<ulid>` with the disk path on the target.
5. Target peer's `compute_provision` registers the disk with its local libvirtd pool, defines the VM (same XML), starts it.
6. Source peer publishes `compute/inventory/<source-peer>` with the VM removed.
7. Workbench panel updates live.

## 10. Birthright integration

No new Birthright step required. `compute_provision` is fully automatic:

- `libvirtd.socket` is already enabled by the RPM `%post`.
- The `mde-vms` storage pool is created on first `compute/create` if absent.
- VM subnet route (`10.42.128.0/17`) is pushed to all peers via mesh-storage when the first VM on the mesh is created.
- Cockpit + cockpit-machines is listed in the RPM `Requires` and accessible from the Workbench panel's `[Open in Cockpit]` button.

## 11. Dependencies

| Dependency | Reason |
|---|---|
| MESHFS-* complete | virtiofsd shares require `/mnt/mesh-storage` mounted |
| BUS-1 broker | All compute Bus topics require the broker |
| INST-* | libvirtd + cockpit-machines installed via RPM `%post` |
| CA peer designation | `cert_authority` worker requires the CA key on the CA peer (set at Birthright) |

---

## 12. Out of scope for VIRT-1..12

- Live migration (no shared storage; cold migration is the only supported path)
- VM templates library / image catalog (post-5.0 VIRT follow-on)
- VM console (Cockpit-machines handles this via the `[Open in Cockpit]` deep-link)
- GPU passthrough (post-5.0)
- Container-native Nebula enrollment (containers share host peer's overlay IP; L2 applies to KVM VMs only)
