## 0. Why federate?

The platform's identity is **one mesh, one passcode, flat trust**
(per `[[project_open_mesh_directive]]`). Federation is the
**controlled exception** for when two operators want their
otherwise-independent meshes to subscribe to each other's
audit/event streams without dissolving the single-passcode model
inside either mesh.

Federation is **strictly opt-in + symmetric**. There is no
unilateral peering. There is no asymmetric subscribe (mesh-A
listens to mesh-B without mesh-B agreeing). There is no
default-publish (mesh-A can't publish into mesh-B's topic tree
without mesh-B granting that surface explicitly).

The reference operator deployment is a household with one mesh
that wants to subscribe to a workplace mesh's `oncall/*` alerts,
or two collaborators with independent meshes who want their
`portal/peer-presence` topics visible across the boundary.

## 1. Out-of-band passcode protocol

Federation pairing uses **two operator-typed passcodes** —
one minted by each mesh, exchanged out of band (in person,
voice call, signal session, etc. — explicitly NOT over either
mesh's own channels).

**Mint:**

    mde-bus federation mint-passcode

prints a 6-word BIP-39 mnemonic (e.g. `apple bridge clutch
dance echo fern`). The passcode is bound to:

- The mesh's CA fingerprint (so a passcode from one mesh
  can't be reused against another).
- A 24-hour expiry stamped in the passcode envelope (the
  envelope is a signed CBOR blob the operator never sees;
  the mnemonic is the index into a local `~/.local/share/mde/
  bus/federation-mints/<ulid>.cbor` store).
- A single-use flag (consuming the passcode at the other
  end retires it; another mint requires another invocation).

The passcode is mesh-internal once minted — any peer in the
mesh can `federation mint-passcode` against any other peer
(the lighthouse-elected QNM-Shared leader signs the envelope
via mackesd). The minting peer is operator-known but not
load-bearing for the protocol; the binding is to the mesh CA,
not to the minting peer.

**Exchange:**

Operator-A reads passcode-A out loud (or over a side channel)
to operator-B; operator-B reads passcode-B back. Each
operator types the other's passcode into their mesh's
Workbench:

    Workbench → Mesh → Federation → Accept Pair → [type passcode]

The Workbench surface lifts under TUNE-15.b.

**Consume:**

The receiving daemon validates: passcode signature against the
remote CA fingerprint embedded in the envelope; expiry not
exceeded; mnemonic not already consumed in
`~/.local/share/mde/bus/federation-consumed/<ulid>.cbor`.

A valid consume triggers the CA cross-sign protocol (§3).

## 2. Symmetric subscribe-only grant model

Once paired, the **default grant is subscribe-only in both
directions**. Mesh-A peers can subscribe to mesh-B topics; mesh-B
peers can subscribe to mesh-A topics. Neither side can publish
across the boundary by default.

**Grant scope:**

The pairing creates two symmetric `federation grant` records,
one per mesh, stored at `~/.local/share/mde/bus/federation.yaml`:

    pairs:
      - peer-mesh-id: "<mesh-B-CA-fingerprint>"
        peer-mesh-label: "Workplace"
        established: 2026-05-27T14:32:00Z
        established-by-passcode-ulid: <ulid>
        subscribe-topics: ["#"]      # default: subscribe everything
        publish-topics: []           # default: publish nothing
        excluded-topics:
          - "passcode/*"             # never federate auth state
          - "federation/*"           # no recursive federation
          - "clipboard/*"            # privacy default
          - "voip/presence/*"        # privacy default
          - "input/*"                # privacy default

**Operator-explicit publish upgrade:**

To grant publish rights into mesh-B for a specific topic tree:

    mde-bus federation grant-publish <peer-mesh-id> <topic-pattern>

Both meshes must run the symmetric command — A grants B
permission to publish into A's tree; B grants A the same. The
symmetric handshake means no mesh can be unilaterally written
to without its operator approving.

The audit log (§5) records every grant change with operator
intent + timestamp + the symmetric grant on the other side.

## 3. Nebula CA cross-sign protocol

Federation requires peers from mesh-A to authenticate as valid
to mesh-B's Nebula overlay. The cross-sign protocol:

1. **Exchange:** During the OOB passcode consume (§1), each
   mesh's CA exchanges signed certificate-chains via the
   subscribe-only Nebula bridge that's about to be established.
2. **Sign:** Mesh-A's CA cross-signs mesh-B's root CA cert as a
   trust-anchor with a scope-limited capability:
   `purpose: federation-subscribe; expires-at: <pair.established + 1 year>;
    constrained-topics: <pair.subscribe-topics minus excluded>`.
3. **Install:** Each mesh's nebula-supervisor (`crates/mackesd/
   src/workers/nebula_supervisor.rs`) installs the cross-signed
   cert into `/etc/nebula/federation-trusts/<peer-mesh-id>.crt`
   (mode 0600 root).
4. **Rotate:** A 1-year expiry forces re-pairing annually.
   `mde-bus federation rotate <peer-mesh-id>` regenerates the
   cross-sign without re-running the OOB passcode exchange
   (the original passcode-envelope's mesh-binding survives).
5. **Revoke:** Either side can run `mde-bus federation revoke
   <peer-mesh-id>` to nullify the trust + delete the cert; the
   other side observes the revocation via Bus topic
   `federation/revoked/<peer-mesh-id>` and removes its own
   side symmetrically.

The cross-sign capability is **narrower than full peer
membership** — federated peers cannot mint mesh-internal
passcodes, cannot read mesh-internal `passcode/*` topics, and
cannot dial GFS bricks (mesh-home stays inside the home mesh).
Federation is a **publish/subscribe overlay**, not a full
peer-trust extension.

## 4. Workbench accept-pair UI (TUNE-15.b)

Under `crates/mde-workbench/src/panels/mesh/federation/`:

- **Mint pane:** "Generate passcode for sharing" → 6-word
  mnemonic + countdown (24 h expiry) + revoke-before-use button.
- **Accept pane:** "Type passcode from peer mesh" → 6-word
  input field with BIP-39 autocomplete + on-success transition
  to the grant-tuning pane.
- **Grant-tuning pane:** subscribe-topics multiselect (default
  `#` minus excluded) + publish-topics empty by default with
  explicit "Add publish grant" button per topic-pattern.
- **Active-pairs pane:** list of established pairs with peer-
  mesh-label, established date, expiry countdown, revoke
  button, rotate button, audit log link.

The UI does NOT expose the CA cross-sign step — it's
automatic on pair-acceptance.

## 5. Federation audit log

Every federation lifecycle event lands as a Bus message on
topic `federation/<event-type>/<peer-mesh-id>` with a CBOR
payload:

    {
      "ulid": "01HMZJK...",
      "event": "pair-established",
      "peer-mesh-id": "<fingerprint>",
      "peer-mesh-label": "Workplace",
      "passcode-ulid": "<consume-ulid>",
      "by-operator": "<local-peer-id>",
      "subscribe-topics": ["#"],
      "publish-topics": [],
      "excluded-topics": ["passcode/*", ...]
    }

Event types: `pair-established`, `pair-revoked`,
`grant-publish-added`, `grant-publish-removed`,
`grant-subscribe-narrowed`, `pair-rotated`,
`pair-expired-warning` (T-30d, T-7d, T-1d).

These messages persist into the standard audit topic
(BUS-7.1) + display in the Workbench Mesh Activity pane.

## 6. Federated peers + 8-peer cap exemption (TUNE-15.d)

Federated peers do NOT count against the Q22 8-peer cap. The
8-peer cap is a property of a single mesh's flat trust set;
federation is an inter-mesh bridge, not membership.

The Peer Card render (mde-peer-card) gains:

- **Distinct icon variant:** standard peer icon overlaid with
  a small bidirectional-arrow chip in the bottom-right (the
  federated-peer glyph).
- **"External mesh" badge** above the hostname, showing the
  peer's mesh-label (e.g. "Workplace") in subdued grey.
- **Subscribe-only or two-way indicator** in the corner of the
  card: down-arrow only if subscribe-only; double-arrow if
  symmetric publish grants exist.

## 7. Acceptance criteria (TUNE-15 closure rolls these up)

The TUNE-15 epic ships when **all** of these are operator-
confirmed:

1. `mde-bus federation mint-passcode` prints a 6-word BIP-39
   mnemonic + writes the envelope to `~/.local/share/mde/bus/
   federation-mints/`.
2. Pairing across two real Nebula meshes succeeds end-to-end:
   passcode exchange → consume → CA cross-sign → first
   subscribe round-trips within 30 s.
3. `~/.local/share/mde/bus/federation.yaml` reflects both
   pairs after handshake; subscribe-topics default `#` minus
   excluded.
4. `mde-bus federation grant-publish <peer> portal/peer-presence/*`
   on both sides enables publish in that scope; non-granted
   topics rejected with audit entry.
5. Workbench accept-pair UI completes the round-trip without
   CLI fallback.
6. `federation/pair-established/<peer>` Bus message lands in
   the audit topic; replay shows it on both sides.
7. Federated peers render with the federation icon variant +
   external-mesh badge in mde-peer-card; don't count against
   the Q22 cap.
8. `mde-bus federation revoke <peer>` deletes both sides'
   cross-sign certs + emits revocation events.
9. AI_GOVERNANCE.md §11 1.0 roadmap item 1 gains
   "BUS-7.7-FED complete" sub-bullet.

## 8. Risks

| Risk | Mitigation |
|---|---|
| OOB passcode shoulder-surfed | 24-hour expiry; single-use; envelope-bound to mesh CA so a leaked mnemonic can't be reused against another mesh |
| Federated peer impersonates mesh-internal peer | CA cross-sign is scope-limited (`purpose: federation-subscribe`); nebula-supervisor refuses mesh-internal capabilities from federation-trust certs |
| Recursive federation creates trust loops | `federation/*` topics in the default-excluded list; explicit operator-typed publish grant required to break it (and even then, the receiving side's `federation/*` filter still applies) |
| Federation drift between meshes (one side revokes, other doesn't notice) | Bus topic `federation/revoked/<peer>` propagates; each side's worker removes the local cert + grant on receipt; 1-year cert expiry forces re-pairing |
| Operator misclicks "grant publish" on sensitive topic | Workbench grant-publish pane requires typing the topic pattern explicitly + a confirmation modal; the audit log makes mistakes recoverable |

## 9. Out-of-scope (v1.0)

- **Multi-hop federation** (mesh-A → mesh-B → mesh-C transitive).
  v1.0 ships only direct one-hop pairs.
- **Asymmetric pairing** (mesh-A subscribes to mesh-B without
  mesh-B subscribing to mesh-A). Both sides always pair
  symmetrically; the publish-grant model is what carries
  asymmetry within a symmetric pair.
- **Federation with non-Mackes meshes.** v1.0 is mesh-to-mesh
  Mackes-only; Matrix/ActivityPub/XMPP bridges deferred.
