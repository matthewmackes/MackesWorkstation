## 1. Mental model

### 1.1 Cards-and-breadcrumbs axiom (R5)

Every object in MDE Portal is a **card**. Every navigation step is a **breadcrumb segment**. These are the only two primitive UI surfaces. Cards and segments are the same primitive in different render modes.

**Card schema (12 fields, R5-Q3):**

```json
{
  "id": "<stable-mesh-merged>",
  "type": "app|file|peer|activity|setting|workspace|tag|tray|zone|contact|container|...",
  "name": "...",
  "icon-uri": "...",
  "color-tag": "...",
  "parent-tag(s)": ["..."],
  "state": { "read": false, "starred": false, "pinned": false, "tags": [...] },
  "actions": [{"label": "...", "primary": true, "...": "..."}],
  "description": "...",
  "source-peer": "<peer-uid>",
  "ts-modified": "<iso8601>",
  "body-text": "...",
  "children": [/* recursive Card[] for composite cards */],
  "schema_version": 1
}
```

**Card identity (R5-Q9):** Stable IDs are derived from the underlying object — apps = `.desktop-id`, files = `<peer>:<path>`, peers = mesh-uid, activities = ULID. Cross-peer view auto-merges identical cards (Firefox-on-laptop and Firefox-on-phone unify).

**Card render modes (6, R5-Q2 + R5-Q7 + R5-Q17 + R5-Q21):**

1. **segment** — compact, mono, chevron-bordered (breadcrumb-segment context)
2. **cascade-card** — full icon + name + 1-line description + Actions button (Hub default)
3. **list-row** — dense single-line (Library long lists, notifications popover)
4. **mini-tree-cell** — chevron-bordered cell with number + name + thumbnail (workspaces, tray icons)
5. **lock-widget** — special lock-screen rendering for clock/weather/battery/etc.
6. **hero** — full-bleed 3x-cascade-card-height poster (enriched-card default; tall poster with artwork + title + Wikipedia snippet + 'Read more' chip)

**Cross-mode animation:** shared-element morph + spring physics overshoot (~250ms). Cards feel like persistent objects, not redraws (R5-Q8).

**Card composition (R5-Q10):** Cards have an optional `children: Card[]` field. Tree-of-cards is the canonical data structure. Cascade columns render `parent → child` drills.

### 1.2 Three-scale Portal model (R3-Q50)

The user-facing surface is ONE THING at three scales:

| Scale | Trigger | Form |
|---|---|---|
| **Dock** | Always-visible | 56px bottom strip; horizontal breadcrumb |
| **Portal-compact** | Mod+Space (or Mod+\\ on IBus systems, R6-Q7) | 336px-tall floating strip; mesh-glance globe |
| **Portal-full** | Super-tap | Full-screen scratchpad shell; 5 layers |

Dock↔Portal-full transition is an **element-by-element morph** (R3-Q49): each Dock element animates to its corresponding position in Portal; the Dock IS Portal in collapsed form.

### 1.3 Wayland surfaces (6 total)

mde-portal owns six Wayland surfaces in a single Rust process:

1. **Dock** (wlr-layer-shell `bottom`, exclusive). 56px tall, per-output (R3-Q1, R3-Q45).
2. **Portal-compact** (wlr-layer-shell `top`, summoned via Mod+Space/Mod+\\). 336px tall.
3. **Portal-full scratchpad** (wlr-layer-shell `top`, summoned via Super-tap; hidden via i3 scratchpad).
4. **Lock screen** (wlr-layer-shell `overlay`, replaces swaylock theming).
5. **Boot/shutdown theater** — full-screen overlay rendering real-time systemd telemetry around an assembling/dimming M watermark (2–3s budget each).
6. **Mesh-wallpaper** (wlr-layer-shell `background`, R6-Q27). 12s default update (6–60s range), auto-pause when fully obscured.

App-switcher **lives on the Dock** (R3-Q41 reverses R2-Q55). Alt+Tab cycles running-app icons with Aero-peek above current. No separate switcher surface.

### 3.4 VoIP layer

Default landing: dialpad + recent calls only — Mac Phone-app minimal landing (R10-Q10). View-tail: `› VoIP:<state>` (R10-Q11).

**SIP architecture (Round 9 + R11 lock 2026-05-26 — Kamailio retired):**

R11 supersedes R9-Q1/Q2/Q4. The Kamailio-per-peer container model is removed
entirely; PJSIP registers directly to a per-peer Vitelity sub-account using
its native multi-account support. Each peer runs **two PJSIP accounts
simultaneously**: a mesh-internal extension and a Vitelity sub-account.

**Birthright deploys zero containers** (R11 clarification 2026-05-26). The
VoIP stack runs as native processes inside `mded`/`mde-portal`. Podman
remains in the architecture exclusively for the user-managed CONTAINER-*
feature surface in Control (operator pulls/runs/manages their own pods +
containers). The **only** container the VoIP stack itself may spawn is an
on-demand `mediasoup` media bridge during 3+ party calls (see VOIP-9) —
spawned by `mded` at call-start, torn down at call-end, never at install
or login.

- **PJSIP as the only SIP stack** — in-mde-portal, no SIP server in the loop (R8-Q86 + R11 lock)
- **Two accounts per peer** (R11): account 1 = `sip:<user>@mesh.mde` for intra-mesh; account 2 = `sip:<sub-account>@sip.vitelity.net` for PSTN. Both register on `mded` startup.
- **Vitelity sub-account creds** stored per-peer in `~/.config/mde/voip/vitelity.toml`, provisioned by Birthright (R11; supersedes R9-Q2 container-secrets model)
- **Bare-hostname dialing** resolves via mesh DNS (`<host>.mesh.mde`, R9-Q3) — intra-mesh INVITE goes peer→peer directly, no relay
- **External calls** route through the dialing peer's own Vitelity link (R11; supersedes R9-Q4 smart-routing). Optional: `mded` broadcasts Vitelity-link RTT via Bus; dialer UI shows the local link's RTT and can suggest "Place via <peer>?" before dialing if a markedly better link exists. No automatic re-route — operator-explicit only.
- **Inbound:** Vitelity-side SimRing across all peers' sub-accounts; each peer's PJSIP reads focus state from Bus and responds **`486 Busy Here`** immediately if `focused_peer != self`. Only the focused peer answers (R11; supersedes R9-Q5 Kamailio fork)
- **Tiebreaker:** when two peers both claim focus, the lower mesh-ULID wins (Bus ordering); the other returns 486
- **Voicemail:** Vitelity voicemail-to-email → mesh-home `~/Documents/Voicemail/<peer>/<iso8601>.mp3` + transcript card (R9-Q6, unchanged)
- **Video:** VP9 default + H.264 fallback + AV1 opt-in + SRTP + adaptive 90p–1080p (R9-Q7, unchanged)
- **Call UI:** Portal-full overlay (R9-Q8, unchanged)
- **Group calls:** PJSIP can't mix N participants, so calls >2 spawn an **ephemeral media bridge** (mediasoup) on the best-uplink peer for the duration of the call; signaling coordinated via Bus, bridge container torn down on last-leave (R11; supersedes R9-Q9 Kamailio focus). 8-participant cap preserved.
- **Recording:** per-peer toggle in Customize, off by default (R9-Q10, unchanged)
- **SMS:** Vitelity API → cards in calls/ activity, inline replies, mesh-synced threads (R9-Q11, unchanged)
- **Contacts:** universal cards (type='contact') in mesh-home (R9-Q12, unchanged) — fully participate in tag system
- **Audio routing:** PipeWire role-based + Mod+Space PTT during calls (context-sensitive, R9-Q13, unchanged)
- **Echo cancellation + noise suppression:** RNNoise + WebRTC AEC3 (R9-Q14, unchanged)
- **Adaptive bitrate cascade** with 'connection adapting' chip (R9-Q15, unchanged)
- **5 ringtones + per-contact override** (R9-Q16, unchanged)
- **Encryption:** Nebula intra-mesh + SRTP-AES external (R9-Q17, unchanged)
- **Multi-mesh federation:** SIP-over-Matrix bridges (R9-Q18) — **deferred past 1.0** under R11; without Kamailio there's no per-peer SIP server to host the bridge. Reinstating means either a single-gateway-peer asymmetry (one peer runs `matrix-appservice-voip`) or a different federation mechanism. Open question.
- **NAT traversal:** Nebula UDP-hole-punching intra-mesh; PJSIP SIP UDP/TLS with rport + STUN for external Vitelity registration (R9-Q19, unchanged)

**Active call edge viz** on globe (R8-Q85): living purple edge + phone-glyph midpoint + audio-sync pulse + 1Hz throb; same for ANY SIP client activity.
