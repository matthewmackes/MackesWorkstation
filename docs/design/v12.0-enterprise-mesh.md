## 8-layer service architecture

```text
┌────────────────────────────────────────────────────────────────┐
│  Layer 8 — GUI panels                                           │
│    workbench/network/mesh_control/* + topology renderer         │
│    Reads through mackesd_core::mesh::*                          │
├────────────────────────────────────────────────────────────────┤
│  Layer 7 — Library facade (mackesd_core)                        │
│    Public Rust API the panel links in-process. No IPC.          │
├────────────────────────────────────────────────────────────────┤
│  Layer 6 — Service traits (service/)                            │
│    NodeService · ConfigService · TelemetryService · PolicyService│
├────────────────────────────────────────────────────────────────┤
│  Layer 5 — Reconciliation engine (reconcile/)                   │
│    Drift detector · State machine · Auto-repair · Retry/backoff │
├────────────────────────────────────────────────────────────────┤
│  Layer 4 — Domain logic (topology/, policy/, validation/)       │
│    Peer-relationship calculator · policy DSL · schema validation│
├────────────────────────────────────────────────────────────────┤
│  Layer 3 — Telemetry ingest (telemetry/)                        │
│    Heartbeat aggregation · link metrics · event-log hash chain  │
├────────────────────────────────────────────────────────────────┤
│  Layer 2 — Persistent store (store/)                            │
│    SQLite WAL · revision tables · audit log · sqlx migrations   │
├────────────────────────────────────────────────────────────────┤
│  Layer 1 — Process supervisor (deploy/)                         │
│    Leader election · systemd unit · CLI · health/metrics export │
└────────────────────────────────────────────────────────────────┘
```

Each layer imports only from layers below it. The library facade
(layer 7) is the only externally-visible surface; everything else
is `pub(crate)` or `pub(super)`. **No networked API exists** — peer
communication happens through the shared filesystem only (heartbeat
JSON files under `~/QNM-Shared/<peer>/mackesd/`).

## 7 state buckets

The SQLite store holds seven distinct kinds of state. Each has its
own table with a `revision_id` + `created_at` + `applied_at`:

| Bucket               | Purpose                                       |
|----------------------|-----------------------------------------------|
| `desired_config`     | What the operator *wants* the mesh to be.     |
| `runtime_state`      | What `mackesd` is currently *trying* to make happen (the in-flight transitions). |
| `observed_telemetry` | What every peer *actually reports* about itself + its links. |
| `calculated_topology`| The pure-function output of `desired_config` + node policies — expected peer adjacencies + route tables. |
| `pending_changes`    | Drafted but unapproved revisions, waiting in the operator's inbox. |
| `applied_changes`    | Successful deploys — durable history with full snapshot retention for rollback. |
| `failed_changes`     | Deploys that broke during validation or apply — the failure reason chain is captured for debugging. |

Every write to any bucket is a single SQL transaction (per
12.2.3); partial-applied states never persist.

## Deployment lifecycle state machine

```text
                  ┌────────┐
                  │ Draft  │  ← operator edits desired_config
                  └───┬────┘
                      │ submit
                      ▼
              ┌───────────────┐    ┌──────────────────┐
              │   Validated   │───▶│ FailedValidation │
              └───────┬───────┘    └──────────────────┘
                      │ approve
                      ▼
              ┌───────────────┐
              │   Approved    │
              └───────┬───────┘
                      │
                      ▼
              ┌───────────────┐    ┌──────────────────────────────┐
              │   Deploying   │───▶│ FailedDeployment → RolledBack │
              └───────┬───────┘    └──────────────────────────────┘
                      │
                      ▼
              ┌───────────────┐
              │    Applied    │
              └───────┬───────┘
                      │ reconciler verifies runtime matches desired
                      ▼
              ┌───────────────┐
              │   Verified    │
              └───────────────┘
```

Persisted in `applied_changes` (for Verified rows) / `failed_changes`
(for FailedValidation and RolledBack rows). The state machine itself
runs in the reconcile loop (`reconcile/` module).

## Leader election

Every peer runs `mackesd`. The leader is the only writer to the
shared `desired_config` mirror. Election is filesystem-based:

1. On startup, `mackesd` attempts to acquire
   `~/QNM-Shared/.mackesd-leader.lock` (an `fs2`-backed advisory
   file lock with a 60-second lease).
2. The lease is renewed every 20 seconds. On miss, the next peer
   in lexicographic `node-id` order acquires the lease.
3. **Split-brain detection:** on lease conflict (two peers both
   think they're leader), the side with the older
   `applied_revision` yields and re-hydrates from the shared store.
4. **Fencing:** a deposed leader marks its in-memory state stale
   and re-reads the store before serving reads again.

## Library surface (`mackesd_core`)

Stable public API the panel imports. Snapshot-tested via `insta`
so any change to the consumed surface fails CI loudly (per 12.11.5).

```rust
pub fn nodes() -> Vec<NodeView>;
pub fn topology() -> TopologySnapshot;
pub fn links() -> Vec<LinkHealth>;
pub fn revisions() -> Vec<RevisionView>;
pub fn diff(a: RevisionId, b: RevisionId) -> RevisionDiff;
pub fn pending() -> Vec<PendingChange>;
pub fn audit_chain() -> Vec<EventRow>;
pub fn healthz() -> HealthReport;
```

All read functions return owned value types — no internal
references escape. Mutations land through the CLI (`mackesd
enroll | decommission | rollback | rotate-passcode`) or via the
panel's "Approve" / "Reject" buttons on pending changes.

## Why no networked API

User lock 2026-05-19, item 3 of the 5-question survey:
**Inter-component access is in-process library + shared filesystem.**

- The panel imports `mackesd_core::*` directly. Function calls,
  not REST.
- Peer-to-peer sync uses `~/QNM-Shared/<peer>/mackesd/` (SSHFS-
  backed via the mesh filesystem). Append-only JSON files, hash
  signed, leader aggregates on its next tick.
- No HTTP server, no TCP listener, no D-Bus service.

The intent: every byte of mesh-control traffic rides the same
WireGuard tunnel + SSHFS overlay that mesh-FS already uses.
Operators don't open a new firewall hole, and external attackers
can't reach the control plane without compromising mesh trust.
