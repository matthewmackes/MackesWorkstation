# v2.6 — Wayland VNC server (wayvnc)

**Status:** Locked 2026-05-24
**Target release:** v2.6 (first non-major minor after v2.5 +
v5.0.0)
**Worklist:** `docs/PROJECT_WORKLIST.md` § "RD-1..RD-5: v2.6 —
Wayland VNC server gap"

## 1. The gap

`v2.0.0` hard-switched the session host to **sway** (Wayland-
only — see `~/.claude/.../project_v2_0_0_mackes_de.md`), but
`mackes/birthright.py:apply_remote_desktop` still installs
`x11vnc` and enables `x11vnc@:0.service`. `x11vnc` mirrors an
X11 `:0` display; on a Wayland-only session there IS no `:0`
and the unit silently fails to bind.

The RPM spec's `Requires: xrdp` + `xrdp-selinux` covers the
RDP path (xrdp ships an Xorg-fallback session that works
under Wayland greeters), but VNC has no equivalent fall-back.
The per-peer **[VNC]** button in
`crates/mde-workbench/src/panels/remote_desktop.rs` shells
`remmina -c vnc://<host>:5900` against a port nothing is
listening on. Discovered 2026-05-24 during the remote-access
capability audit.

## 2. The lock — wayvnc

| | wayvnc | gnome-remote-desktop |
|---|---|---|
| Wayland-native | yes (wlroots screencopy) | yes (PipeWire) |
| RPM closure | ~200 KB | ~30 GNOME packages |
| GNOME dep drag | none | mutter components conflict w/ sway |
| Auth model | Ed25519 keys (PAM-free) | PAM + GNOME keyring |
| Built for sway | yes (Drew DeVault upstream) | no (GNOME-first) |
| Pixel-perfect | yes (per-output capture) | partial (frame-by-frame) |

**Verdict:** wayvnc. The gnome-remote-desktop closure brings
mutter components that conflict with sway's compositor surface
ownership — operator-debuggable but not worth ~30 packages of
GNOME drag for an alternative auth model.

## 3. Architecture

### 3.1 Process model

```
                ┌─────────────────────────────────────┐
                │  Per peer (Wayland-only host)        │
                │                                      │
  swaywm  ──────► compositor (wlroots-based)           │
                │   ↑                                  │
                │   │ wlr-screencopy protocol          │
                │   │                                  │
  mde-wayvnc ────► binds <nebula-overlay-ip>:5900     │
                │   ↑                                  │
                │   │ TLS 1.3 over Nebula overlay      │
                │   │                                  │
  remmina ──────► vnc://<peer.mesh>:5900               │
                │                                      │
                └─────────────────────────────────────┘
```

`wayvnc` runs per-user (via `mde-wayvnc.service`), binds to
the Nebula overlay IP only, and authenticates clients via
Ed25519 public keys synced through the existing
`~/QNM-Shared/<peer>/mackesd/` object store.

### 3.2 Bind boundary

Per the existing NF-17.5 pattern (xrdp on the overlay), wayvnc
must bind to the Nebula overlay address — never the underlay.
The systemd unit reads `/var/lib/mackesd/nebula/overlay-ip`
(GF-1.3.a) and passes it as the `--listen-address` arg.

If `overlay-ip` is missing (pre-enroll), wayvnc fails to start
with a clear `EADDRNOTAVAIL` message — `systemctl --user
status` surfaces it.

### 3.3 Auth model — Ed25519 keypair

Per-peer Ed25519 keypair generated at birthright time, mirroring
the existing `mesh_ssh` layer A pattern:

* **Private key:** `~/.config/mde/wayvnc/identity.key` (0600).
* **Public key:** published to
  `~/QNM-Shared/<self>/mackesd/wayvnc-pubkey.pem` (0644) so the
  QNM-Shared replicator distributes it to every other peer.
* **Authorized keys:** `/etc/wayvnc/authorized_keys.pem`
  rebuilt by the wayvnc supervisor whenever a new peer enrolls
  — concatenation of every `<peer>/mackesd/wayvnc-pubkey.pem`
  in QNM-Shared.

`wayvnc`'s `config` references both files:

```
private_key_file=/home/<user>/.config/mde/wayvnc/identity.key
authorized_keys_file=/etc/wayvnc/authorized_keys.pem
enable_pam=false
```

PAM is explicitly disabled — the Nebula overlay + Ed25519 chain
IS the auth boundary. Per `[[project_open_mesh_directive]]`,
trust within the mesh is flat: anyone who can reach the
overlay AND holds an enrolled key can connect. A future
revocation lands as removal from `authorized_keys.pem` via the
same QNM-Shared rebuild path.

**Acceptance:** an unenrolled host on the overlay (somehow
holding the overlay IP but no enrolled Ed25519 key) cannot
connect. Verified by a bench drill where the operator removes
peer-B's pubkey from peer-A's `authorized_keys.pem`, restarts
wayvnc, and observes that peer-B's VNC attempts get rejected
at the TLS handshake.

## 5. Out of scope

* **Audio over VNC.** wayvnc 0.x has no audio channel; that's
  upstream's roadmap, not ours. RDP via xrdp covers the audio
  case for now.
* **Cursor synchronization across multiple displays.** wayvnc
  is per-output (`--output=<name>`) — multi-monitor sessions
  need a tool like Sunshine + Moonlight, which is a larger v2.7
  pick.
* **Per-peer policy beyond authorized_keys.** Open-mesh / flat-
  trust per the standing directive; no ACLs.

## 6. Migration from v2.5

* `dnf upgrade` swaps `x11vnc` → `wayvnc` (the spec's RPM
  closure pulls the new dep + removes the old).
* `%post` disables `x11vnc@:0.service` if present, then
  enables `mde-wayvnc.service` once birthright's
  `apply_remote_desktop` runs.
* Existing operator VNC sessions die at upgrade time (x11vnc
  is replaced); next session uses wayvnc + the new Ed25519
  auth.
* Operators get a one-time "regenerate Ed25519 keys + re-pair"
  prompt the first time they open Workbench → Remote Desktop
  post-upgrade.
