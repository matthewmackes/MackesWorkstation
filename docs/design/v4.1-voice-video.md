# v4.1 — Voice & Video over Mesh (with Vitelity PSTN edge)

**Status:** Plan locked 2026-05-23. Target release: **v4.1.0**.
**Companion files:**
- `docs/PROJECT_WORKLIST.md § v4.1.0 Voice & Video epic` (actionable
  task list — VV-1 through VV-15).
- `docs/design/v12.0-enterprise-mesh.md` (mesh control plane this
  builds on top of).
- `docs/design/v12-connectivity-scope.md` (the 16-peer, 50%-LAN /
  50%-WAN audience these decisions are tuned for).

This is a planning document, not an implementation. Per
`.claude/CLAUDE.md § 0.5` no code lands until the operator says
"execute" or `cut release`.

---

## 1. Goal

Bring full real-time voice + video + presence + chat to MDE, with
seamless reach across the mesh AND out to the PSTN, so an operator
can:

1. **Click any peer in the Workbench and talk / video-call / chat** —
   no manual SIP config, no firewall holes, no third-party SaaS.
2. **Place a call from any peer through any peer when direct
   reach is blocked**, using mesh transit-routing to pick the
   highest-quality path (the same throughput-wins policy locked
   in v12-connectivity-scope Q23).
3. **Dial public phone numbers from any peer** through that peer's
   own Vitelity Communications SIP trunk. Inbound DIDs ring the
   peer they belong to.
4. **Operate a "dual plan"** — one policy bucket controls the
   in-mesh voice topology, a second controls each peer's public
   PSTN edge. Both deploy through the existing Phase-12 draft →
   validated → approved → applied → verified state machine.

## 2. Audience + scope locks (inherited from v12)

Unchanged from the connectivity-scope survey:

- **Peer cap:** ≤16 (Q2).
- **Geography:** single region (Q4) — sub-100 ms RTT between any
  two peers via the best path.
- **Same-LAN ratio:** ~50% (Q3) — half of peer-pairs share a
  broadcast domain; the other half must reach via relay /
  HTTPS-443 fallback.
- **Headless first-class:** (Q5) — the Pi-class Host peer must
  serve voice the same as a GUI peer. The embedded client is
  optional on a peer; the per-host Asterisk daemon is not.
- **Platform:** Linux desktop only (Q21). Android / iOS softphone
  parity is **explicitly out of scope** — third-party SIP clients
  (Linphone, Zoiper, Acrobits) can register against the mesh by
  pointing at a peer's local Asterisk, but MDE doesn't ship a
  phone app.

## 3. Six-layer architecture

```text
┌──────────────────────────────────────────────────────────────┐
│  Layer 6 — Workbench Voice surface (mde-voice)               │
│    Iced crate: Calls · Contacts · Rooms · Voicemail ·        │
│    Presence · History · Vitelity Account                     │
├──────────────────────────────────────────────────────────────┤
│  Layer 5 — Embedded SIP/video client (PJSIP via FFI)         │
│    Registers to 127.0.0.1:5061 (local Asterisk).             │
│    PipeWire capture + playback. VP8/Opus by default.         │
├──────────────────────────────────────────────────────────────┤
│  Layer 4 — Vitelity PSTN edge (per-peer, not centralized)    │
│    Each peer's Asterisk REGISTERs its own Vitelity           │
│    sub-account. Inbound DIDs ring the owning peer.           │
├──────────────────────────────────────────────────────────────┤
│  Layer 3 — Mesh dialplan + transit routing                   │
│    Extension scheme: 1NNN per node_id. Local dial → direct.  │
│    Unreachable peer → B2BUA bridge through the best transit  │
│    peer (consults mackesd connectivity telemetry).           │
├──────────────────────────────────────────────────────────────┤
│  Layer 2 — Per-host Asterisk daemon (mde-voice-asterisk)     │
│    asterisk-mde.service — chan_pjsip, ConfBridge, MoH,       │
│    voicemail, MESSAGE, SIMPLE presence. Bind: lo + nebula1   │
│    only (defense-in-depth — never on a public iface).        │
├──────────────────────────────────────────────────────────────┤
│  Layer 1 — Config generation (mde-voice-config)              │
│    Pure-function: desired_config + observed_telemetry →      │
│    pjsip.conf + extensions.conf + confbridge.conf + voice-   │
│    mail.conf. Plugs into mackesd's policy dispatcher.        │
└──────────────────────────────────────────────────────────────┘
```

Each layer imports only from layers below it. Layers 1–4 ship on
every peer including headless. Layers 5–6 ship only on GUI peers
(the embedded client is launched via the existing
`mde --voice` entry point; headless peers skip it entirely).

## 4. Per-host Asterisk daemon (Layer 2)

### 4.1 Distribution

- **Package:** `asterisk` 21.x LTS from Fedora 44's official repo
  (currently `asterisk-21.4.x`). Pulled in by
  `packaging/fedora/mackes-shell.spec`'s `Requires:`. No
  third-party Sangoma repo — the F44 build covers chan_pjsip,
  ConfBridge, app_voicemail, and the MESSAGE channel driver.
- **Service unit:** `data/systemd/asterisk-mde.service` — a
  drop-in *not* the upstream `asterisk.service`. Runs as a
  dedicated `_asterisk_mde` user with its own
  `/var/lib/asterisk-mde/` data root so any pre-existing
  operator Asterisk install on the box (a rare case but
  possible) keeps its config untouched.
- **CLI namespacing:** `mde voice cli` → `asterisk -rx 'pjsip
  show endpoints'` against the MDE socket. The bare `asterisk
  -r` continues to target the upstream daemon if present.

### 4.2 Binding

Asterisk binds to two interfaces only:

1. `127.0.0.1` — for the embedded SIP client.
2. The Nebula mesh interface (`nebula1` — Nebula's default tun
   device name on Linux; the v2.5 NF-* sweep replaced the v1.x
   WireGuard transport with Nebula, so the original plan's
   `wg-mesh` reference is stale and was updated 2026-05-24) —
   for inter-peer SIP signaling + RTP.

The Vitelity REGISTER session is **outbound only** — Asterisk
opens a connection to `out.vitelity.net:5061` (TLS); Vitelity
sends inbound INVITEs back over the established session (per
RFC 5626 / outbound proxy). **No port 5060 is exposed on any
public interface, ever.**

### 4.3 Endpoint identity

Each peer has exactly **two** PJSIP endpoints in its local
`pjsip.conf`:

| Endpoint              | Used by                                  |
|-----------------------|------------------------------------------|
| `mde-local`           | the embedded SIP/video client            |
| `mde-vitelity-trunk`  | this peer's Vitelity sub-account         |

Plus a generated AOR per **remote** peer for mesh dial-out:

| AOR pattern         | Maps to                                  |
|---------------------|------------------------------------------|
| `1NNN@wg-<node_id>` | the remote peer's `mde-local` endpoint   |

The `1NNN` extension is the 4-digit form of the lexicographic
index of the peer's `node_id`. With a 16-peer cap, extensions
run `1001`–`1016`. Generated; not operator-editable. Operators
attach **display names** (e.g., "Matthew laptop") via the
Workbench; the dialplan stays numeric.

## 5. Mesh dialplan + transit routing (Layer 3)

### 5.1 Direct dial

`extensions.conf` includes a generated `[mde-mesh]` context.
Dialing `1003` from the embedded client:

```text
exten => _1XXX,1, NoOp(Mesh dial to ${EXTEN})
 same  => n, Set(TARGET=${MDE_MESH_LOOKUP(${EXTEN})})
 same  => n, GotoIf($["${TARGET}" = ""]?notfound)
 same  => n, Dial(PJSIP/${EXTEN}@${TARGET}, 30)
 same  => n, Hangup()
```

`MDE_MESH_LOOKUP` is a custom AGI script
(`/usr/libexec/mde-voice/mesh_lookup.py`) that consults
`mackesd_core::voice::best_path(target_ext)` and returns:

- **Empty string** → unknown extension → fail with "Number
  not in directory" announcement.
- **`wg-<node_id>`** → direct path is reachable; SIP INVITE
  goes straight to the destination peer's Asterisk over
  WireGuard.
- **`transit-<via_node_id>`** → must hop through another peer;
  trigger B2BUA bridge (see 5.2).

### 5.2 Transit (B2BUA bridge)

When `MDE_MESH_LOOKUP` returns a transit path, the dialplan
opens a back-to-back-user-agent leg:

```text
 same  => n, Set(VIA=${CUT(TARGET,-,2)})
 same  => n, Dial(PJSIP/transit-relay@${VIA}, 30,
                  b(mde-voice^transit-leg^${EXTEN}))
```

The transit peer's Asterisk receives an INVITE on its
`transit-relay` endpoint, looks up the bridged extension via
the same AGI helper, and re-dials towards the destination.
Net result: two SIP legs, one transparent bridge between them.

**Why B2BUA, not SIP REFER:**

- REFER asks the originating peer to re-INVITE the destination
  directly. That defeats the point — the originating peer can't
  reach the destination, which is why we're transiting.
- B2BUA costs one extra RTT (~5 ms LAN, ~25 ms transit) and one
  extra ICE/SRTP handshake. Both are well below the human
  voice-quality threshold.
- DTLS fingerprints terminate at the transit peer. The transit
  peer can see media metadata (durations, packet sizes) but
  not media content unless it actively MITMs — which the mesh
  trust model already governs (every peer is a trusted member
  of the cloud).

### 5.3 `best_path` heuristic

```text
fn best_path(target: NodeId) -> Path {
    // Inherit the connectivity layer's already-measured paths.
    let candidates = mackesd::connectivity::candidates(target);

    // Voice-specific weight: prefer paths under 80 ms RTT
    // AND under 5% loss. Throughput is not the bottleneck
    // for Opus (24 kbps) or VP8@360p (~600 kbps).
    candidates
        .into_iter()
        .filter(|c| c.rtt_ms < 80 && c.loss_pct < 5.0)
        .min_by_key(|c| c.rtt_ms + (c.loss_pct as u32 * 10))
        .map(|c| Path::Direct(c.via))
        .unwrap_or_else(|| Path::Transit(pick_relay(target)))
}
```

This is intentionally different from the connectivity layer's
default throughput-wins (Q23) policy. **Voice favors latency
over throughput** — a 200 Mbps relay path beats a 30 Mbps direct
path for bulk transfer but loses on a phone call. The
voice-router is a thin overlay that consults connectivity but
makes its own decision.

## 6. Vitelity PSTN edge (Layer 4) — per-peer

### 6.1 Account model

Every peer that wants public-network reach has its own Vitelity
sub-account. From the Vitelity portal:

- Username (per-peer, e.g., `mde-alice-laptop`).
- Password.
- Outbound proxy: `out.vitelity.net:5061` (TLS).
- Inbound DID(s): one or more E.164 numbers Vitelity routes to
  this sub-account.
- Outbound CID: a verified DID owned by the operator (Vitelity
  rejects unverified CID).

These are entered **only** through the Workbench Voice →
Vitelity Account panel (Layer 6 surface). The operator never
edits `pjsip.conf` directly.

### 6.2 Inbound

Vitelity sends an INVITE to the registered peer. The peer's
`extensions.conf` `[mde-vitelity-inbound]` context routes
based on the operator's per-DID rules:

| Rule kind       | Effect                                                |
|-----------------|-------------------------------------------------------|
| `ring-self`     | Ring this peer's embedded client only.                |
| `ring-extn`     | Forward to a specific mesh extension (e.g., 1007).    |
| `ring-group`    | Simultaneous ring of N listed extensions; first wins. |
| `voicemail`     | Go straight to this peer's voicemail box.             |
| `confbridge`    | Drop the caller into a named conference room.         |
| `ivr`           | Custom announcement + DTMF menu (v4.2+ scope).        |

### 6.3 Outbound

`extensions.conf` `[mde-vitelity-outbound]`:

```text
exten => _9NXXNXXXXXX,1, NoOp(PSTN dial: ${EXTEN:1})
 same  => n, Set(CALLERID(num)=${MDE_VITELITY_CID})
 same  => n, Dial(PJSIP/${EXTEN:1}@mde-vitelity-trunk, 60)
 same  => n, Hangup()
```

The operator dials `9` + the public number (US/CA pattern
shown; international is `_900.` with country-specific
rewriting). Vitelity's billing applies per-minute.

### 6.4 Fail-mode

If Vitelity REGISTER fails (auth, network, outage), the peer:

1. Marks its `mde-vitelity-trunk` endpoint as `Unavailable`.
2. Surfaces a non-blocking diagnostics chip on the status
   cluster (per Q12 — subtle, no toast spam).
3. **Does NOT spill PSTN traffic to any other path.** No SIP
   trunk discovery, no public-DERP fallback for PSTN. Either
   the call goes via the operator's Vitelity sub-account or it
   doesn't go.
4. Retries REGISTER with the same gentle exponential backoff
   the connectivity layer uses (5 s → 5 min cap, Q13).

### 6.5 Why per-peer, not one gateway

Operator lock 2026-05-23 (voice-video survey Q2). Tradeoffs:

| Concern                  | Per-peer (locked)            | One gateway                  |
|--------------------------|------------------------------|------------------------------|
| Vitelity sub-account cost| N sub-accounts               | 1 sub-account                |
| Per-peer CID             | Free (each peer picks)       | Shared CID across all calls  |
| Roaming                  | Peer keeps its CID anywhere  | All calls show gateway's CID |
| Gateway outage           | Other peers unaffected       | Whole fleet loses PSTN       |
| DID assignment           | DID directly rings owner     | DID hairpins via gateway     |
| Outbound concurrency cap | Per-peer (each gets N lines) | Whole-fleet shared cap       |

The per-peer model trades a higher Vitelity bill for full PSTN
resilience and direct DID ownership. This matches the broader
mesh design — every peer is equal, no single point of failure.

## 7. Embedded SIP/video client (Layer 5)

### 7.1 Stack

Operator lock 2026-05-23 (Q1): **PJSIP (libpjproject) via
Rust FFI.** Wrapped in two crates:

- `crates/mde-voice-pjsip-sys` — `bindgen`-generated raw
  bindings. Builds against system `pjsip` (Fedora package
  `pjproject-devel`).
- `crates/mde-voice-client` — safe Rust wrapper. Exposes a
  trait surface to Layer 6 (`Call`, `Registration`, `Presence`,
  `MessageSession`, `MediaSink`, `MediaSource`).

The wrapper is async (Tokio) — every PJSIP callback posts a
`Message::Voice(VoiceEvent)` into the Iced app event loop.
The Iced surface never blocks on a SIP operation.

### 7.2 Media

- **Audio capture / playback:** PipeWire via `pw-stream`.
  Reuses the existing v1.1.0 audio compliance work (see
  `docs/design/audio-video-compliance.md`) — no new audio
  plumbing.
- **Video capture:** XDG `org.freedesktop.portal.Camera` portal.
  Works under Wayland, supports the existing camera permission
  prompt. Falls back to `/dev/video0` via v4l2 if the portal is
  unavailable (e.g., headless or unconfigured `xdg-desktop-
  portal-wlr`).
- **Video render:** Iced `image::Handle::from_pixels` with a
  `Subscription` that pulls decoded frames from the wrapper.
  Hardware acceleration deferred — software VP8 decode handles
  360p comfortably on a Pi-class peer.

### 7.3 Codecs

| Direction     | Codec set (priority order)                       |
|---------------|--------------------------------------------------|
| Audio (mesh)  | Opus @ 24 kbps / 48 kHz                          |
| Audio (PSTN)  | PCMU (G.711µ) — Vitelity's preferred             |
| Video (mesh)  | VP8 @ 600 kbps                                   |
| Video (PSTN)  | H.264 baseline via openh264 — disabled by default|

Codec negotiation per-leg. A mesh-to-PSTN call has Asterisk
transcode at the trunk boundary (Opus → PCMU), which is the
only place that costs CPU. Mesh-to-mesh stays in Opus / VP8
end-to-end (no transcode).

### 7.4 Security

- **Signaling:** SIP-over-TLS on every leg. Asterisk's TLS cert
  is the existing mesh CA-issued cert from
  `~/.config/mde/certs/mesh-host.pem`. The Vitelity leg uses
  Vitelity's public CA chain.
- **Media:** SRTP-DTLS on every leg. Key exchange via DTLS
  fingerprint in the SIP/SDP offer.
- **Mesh transport:** all mesh SIP+RTP rides the existing
  WireGuard tunnel — that's defense-in-depth, not the primary
  encryption layer.

## 8. Presence + text chat + streaming (Layer 4 features)

### 8.1 Presence

PJSIP SIMPLE presence (RFC 3856). Every endpoint publishes:

| State      | Trigger                                                |
|------------|--------------------------------------------------------|
| `available`| Embedded client running, no active call                |
| `on-call`  | At least one INVITE in a `Confirmed` dialog            |
| `away`     | Operator idle >5 min (reuses sway-idle timer)          |
| `dnd`      | Operator toggled DND in the Workbench Voice header     |
| `offline`  | Asterisk shows the endpoint as `Unavailable`           |

Subscriptions: every peer's Asterisk SUBSCRIBEs to every
other peer's endpoints. With 16 peers, that's a 16×15=240 subs
mesh — negligible (PJSIP handles thousands). The presence
state surfaces in:

- Peer Card (existing `crates/mde-peer-card`) — small icon
  next to the peer's hostname.
- Workbench Voice → Contacts — full state + last-changed
  timestamp per peer.

### 8.2 Text chat (SIP MESSAGE)

Asterisk's `MessageSend()` dialplan app + the `app_messaging`
module. The embedded client opens a MESSAGE session per peer;
messages persist locally in `~/.local/share/mde/voice/
chat.sqlite` (a new SQLite DB; not shared via mesh-fs because
chat history is per-device by design).

Group chat: a `[mde-room-NNN]` ConfBridge with MESSAGE-only
attendees (no audio). Every peer in the room receives every
MESSAGE. Rooms are created from the Workbench Voice → Rooms
panel.

### 8.3 Streaming audio (MoH + intercom)

Three modes:

1. **Music-on-hold:** standard Asterisk MoH. Operator drops
   audio files into `~/.local/share/mde/voice/moh/`; the
   config generator picks them up.
2. **Intercom / page:** Asterisk's `Page()` app. From the
   Workbench, the operator selects N peers + clicks "Page" —
   their embedded clients auto-answer in speaker mode for
   announcement-only audio.
3. **Conference room:** Asterisk ConfBridge. Workbench Voice →
   Rooms surfaces named rooms (`*81` = `all-hands`, `*82` =
   `huddle-1`, etc.). Each room shows live attendee list,
   mute/unmute, and recording toggle (recording goes to
   `~/.local/share/mde/voice/recordings/`).

## 9. The "dual plan" — policy buckets

Per the existing Phase-12 state model (seven buckets, lifecycle
state machine), Voice adds **two new policy kinds** in
`crates/mackesd/src/policy/`:

### 9.1 `voice_mesh` policy

What it controls:

- Extension-to-peer assignment (default: lexicographic
  `node_id` → `1NNN`; operator can override).
- Display names (`Matthew laptop`, `Office Pi`, etc.).
- Transit-routing preferences (which peers may serve as
  transit relays — defaults to all GUI peers, headless peers
  off by default to keep their CPU free).
- ConfBridge room definitions (name, PIN, recording on/off,
  max attendees).
- MoH playlist selection.
- Presence subscription policy (full mesh by default;
  operator can restrict to named groups).

### 9.2 `voice_public` policy (per peer)

What it controls **on this peer**:

- Vitelity sub-account credentials (username, password,
  outbound proxy URL).
- Owned DID list + per-DID inbound rule.
- Outbound rules (digit-pattern matching, CID, codec
  preference).
- Outbound permission policy — who on this peer may dial PSTN
  (e.g., the local operator only, or any logged-in user).
- Concurrent-call cap (matches the Vitelity plan).

Both policy kinds:

- Have JSON schemas under
  `crates/mackesd/schemas/policy/voice_mesh.json` and
  `voice_public.json`. Schema-validation is the first gate in
  draft → validated.
- Have dispatcher arms in `policy_dispatch::dispatch()` that
  emit Asterisk config-fragment writes accumulating in
  `ReconcileContext`.
- Land through the existing draft → validated → approved →
  deploying → applied → verified lifecycle. Approve / reject
  buttons live in the existing Pending Changes inbox.

The **dual plan** is literal: two policy kinds, two JSON
schemas, two pending-changes review queues, but one lifecycle
state machine. The operator can edit one without touching the
other — change a DID without re-broadcasting a mesh-topology
change.

## 10. Configuration generation flow (Layer 1)

```text
desired_config (voice_mesh + voice_public + nodes)
        │
        ▼
mde-voice-config::generate()        ← pure function
        │
        ├──→ /etc/asterisk-mde/pjsip.conf
        ├──→ /etc/asterisk-mde/extensions.conf
        ├──→ /etc/asterisk-mde/confbridge.conf
        ├──→ /etc/asterisk-mde/voicemail.conf
        └──→ /etc/asterisk-mde/musiconhold.conf
        │
        ▼
asterisk-mde reload                  ← graceful
        │
        ▼
observed_telemetry: ENDPOINT_REGISTERED events
        │
        ▼
mackesd reconciler: Applied → Verified
```

The generator is a pure function (input → output, no I/O), per
the Phase-12 reconciler-dispatcher contract. Tests use golden
fixtures: a `desired_config` JSON in
`crates/mde-voice-config/tests/fixtures/` produces a known set
of `.conf` files, asserted via `insta` snapshot diffs.

## 11. Voice chrome — split into two surfaces (Layer 6)

The original plan put every operator-facing voice screen into
a single Workbench sidebar group `mde-voice`. On 2026-05-24
the operator split that into two surfaces with separate
responsibilities, separate crates, and separate worklist items
(VV-7a + VV-7b). The split is driven by the directive *"the
slide-in interface is for the client for voice and video"* —
the live call/video client is a system-wide overlay, not a
Workbench panel.

### 11.1 VV-7a — Workbench Voice (backend mgmt) — `mde-voice-workbench`

Iced application in the Workbench sidebar (Carbon icon
`phone`). Operator administration / configuration / observation
of the asterisk-mde backend. **No live-call UI lives here.**

| Panel           | Purpose                                                |
|-----------------|--------------------------------------------------------|
| Backend         | `asterisk-mde` health, endpoint table, reload/restart  |
| Vitelity        | This peer's PSTN sub-account, DIDs, rules              |
| Rooms           | ConfBridge room definitions; live attendee list        |
| Voicemail       | Per-peer mailbox; play / delete / mark-read            |
| History         | Call log (CDR-derived); filter by direction / peer     |
| Recordings      | Local recording library; play / delete                 |
| Settings        | Codec preferences, default capture devices, ringtones  |

The **Vitelity panel** is the user's literal ask: an interface
that connects this peer to Vitelity Communications. It surfaces:

1. **Account section** — username + password + outbound proxy
   URL fields. Save triggers a draft `voice_public` policy
   revision; the lifecycle handles validation + apply.
2. **DIDs section** — list of inbound numbers + per-DID rule
   dropdown.
3. **Outbound rules section** — digit-pattern matching with a
   live tester ("if you dial X, the resulting outbound number
   is Y").
4. **CID section** — verified DID picker (Vitelity rejects
   unverified CIDs; the panel hits Vitelity's REST API to
   list verified DIDs for this account).
5. **Status section** — REGISTER state, last successful
   register timestamp, concurrent calls in progress, monthly
   minute count (pulled from Vitelity API).

The **Backend panel** is new vs. the original plan: a
single-page dashboard for the operator running the daemon —
service status, last reload, endpoint registration table,
buffered `asterisk -rx '…'` output, polkit-gated restart /
reload buttons.

Operator action surface — every change goes through draft →
approve → deploy. The panel never writes to `pjsip.conf`
directly; it always submits a `voice_public` policy revision.

### 11.2 VV-7b — Voice/Video Client HUD — `mde-voice-hud`

A wlr-layer-shell overlay anchored to the bottom edge of the
active output. **This is the entire live-call surface.** Slides
up on demand (incoming INVITE, hotkey, or panel applet tap),
slides down when dismissed. Hosts the dialpad, contact picker,
ringing screen, in-call controls (mute / hold / transfer /
DTMF / hangup), live video render with self-view PiP, and
ConfBridge attendee grid.

The HUD is autostarted in the user session so it can answer
incoming calls regardless of which app has focus, and it
honors the active `mde-theme` accent + density tokens. The
visual design bar is "beautiful" — iteration happens through
the `frontend-design` skill with before/after screenshots on
a `ux/UX-VV-7b-*` PR branch per CLAUDE.md §0.11.

The HUD imports VV-6 (`mde-voice-client`) for SIP operations,
VV-8 for PipeWire audio + XDG camera portal, and the new
`mde-mesh-types::voice` presence types so the contact picker
can paint presence dots beside each peer.

A small phone glyph on the mde-panel mesh-status tile gives
the operator a click-to-open affordance from any workspace;
`Super+Space` is the hotkey toggle.

## 12. Acceptance / Definition of Done

A v4.1.0 cut is `[✓]` when:

1. **Two-peer mesh call** — operator at peer A dials `1002`
   from the embedded client; peer B's embedded client rings;
   bidirectional Opus audio + VP8 video for 60 s with no
   gap > 100 ms.
2. **Three-peer transit call** — peer A's direct path to peer
   C is artificially blocked (iptables drop on `nebula1`
   between A and C); A dials `1003`; B's Asterisk B2BUAs the
   call; audio quality on par with direct.
3. **PSTN outbound** — peer A dials `915551234567`; Vitelity
   places the call; the called PSTN endpoint shows A's CID.
4. **PSTN inbound** — caller dials A's owned DID; peer A's
   embedded client rings within 3 s; audio is bidirectional
   PCMU-transcoded-to-Opus.
5. **Presence** — peer B goes idle; peer A's Peer Card shows
   `away` within 30 s.
6. **Chat** — peer A sends MESSAGE to peer B; B's client
   surfaces it within 1 s; history persists across restart.
7. **ConfBridge** — 4 peers join `*81 all-hands`; full mesh
   audio for 5 min; recording produces a single WAV file.
8. **Vitelity outage drill** — `iptables -A OUTPUT -d
   out.vitelity.net -j DROP` on peer A; peer A's
   `voice_public` status flips to `Unavailable` within 60 s
   without spilling traffic; mesh calls continue uninterrupted.
9. **Dual-plan deploy** — operator submits a `voice_public`
   revision (new DID rule) without touching `voice_mesh`;
   only peer A's Asterisk reloads; no other peer's
   `pjsip.conf` rewrites.

## 13. Risks + open decisions

| Risk                              | Mitigation                                  |
|-----------------------------------|---------------------------------------------|
| PJSIP FFI cross-compile pain      | Build against system `pjproject-devel`;     |
|                                   | no static build.                            |
| Asterisk 21 LTS lifecycle on F44  | F45 ships A22 — track the upgrade in       |
|                                   | a separate worklist item when F45 lands.    |
| Vitelity API rate-limit           | Cache verified-DID list per 5-min TTL;      |
|                                   | only refresh on Workbench panel open.       |
| ConfBridge CPU on Pi-class Host   | Transcoding is the cost; ConfBridge runs   |
|                                   | only on GUI peers per transit policy.       |
| Camera portal under sway/wlroots  | Fall back to v4l2 if portal unavailable;   |
|                                   | log + non-blocking diagnostics chip.        |
| openh264 license / distribution   | Disabled by default; operator opts in via  |
|                                   | `voice_mesh.video_codecs += h264`.         |
| First-call latency vs Q8 (≤3 s)   | Pre-register endpoints at boot; INVITE     |
|                                   | path adds ~200 ms for SRTP-DTLS + ICE.     |

Open decisions to revisit at code-time:

- **Recording storage policy** — local per-peer disk only, or
  mesh-fs replicated? Default: local. Replication is a v4.2
  feature.
- **E.911 / emergency-call handling** — Vitelity supports E911
  registration; the Workbench Vitelity panel should require
  an address before activating the trunk. Out of v4.1 unless
  operator legal review surfaces it sooner.
- **IVR support** — out of v4.1 scope; the dialplan generator
  must leave hooks for it (named contexts the operator can
  drop a `[mde-ivr-NNN]` block into) so v4.2 adds the
  panel without a generator rewrite.
